## Product And Data Contracts

### Shopping-List State Matrix

| Existing row | Quantity after add | Checked state | Ordering | Category/icon |
| --- | --- | --- | --- | --- |
| Active, unchecked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Unchecked | Unchanged | Preserve unless request explicitly overrides |
| Active, checked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Reset unchecked; clear `checkedAt` | Move to a newly allocated end position | Preserve unless request explicitly overrides |
| Deleted tombstone | Requested quantity exactly, including `null`; never add stale quantity | Reset unchecked; clear `checkedAt` and `deletedAt` | Move to a newly allocated end position | Preserve unless request explicitly overrides; infer recipe category/icon only when absent |

Recipe batch adds aggregate equal ingredient-reference/unit pairs before mutation: numeric requested quantities sum, while an all-null group remains null. Web, REST, and MCP call the same D1-safe helper.

All shopping-list item writes use an atomic list-level lease/version batch. `ShoppingList` adds `mutationVersion Int @default(0)`, nullable `mutationLease String?`, and `nextSortIndex Int @default(0)`. Every attempt reads the ShoppingList row and observes version `V` first, then reads its items; those reads are sequential and never parallel or items-first. It then submits one non-interactive D1 transaction of parameterized guarded raw-SQL statements: conditionally acquire a random-UUID lease only where `mutationVersion = V AND mutationLease IS NULL`, run every item/create/delete/reorder statement only where that lease exists, allocate the required sort range, then clear the lease and increment version. Because the version read precedes the item read, any concurrent commit visible during or after the item read makes the later version CAS fail; a request can commit only from item state associated with its observed version. If acquisition changes zero rows, every guarded statement is a no-op and the whole request retries from a fresh version-first read. Eight attempts are allowed with fixed delays 5, 10, 20, 40, 80, 160, and 320 ms; exhaustion throws `ShoppingListBusyError`. REST maps it to additive `409 write_conflict` / `Shopping list changed too frequently; retry.`, the web action returns the same 409 message, and MCP returns a retryable tool error with that text. Migration `0024_shopping_list_mutation_coordination.sql` initializes `nextSortIndex` to each list's current `MAX(sortIndex)+1` (zero for empty), version zero, and null lease. Every item mutation uses this coordinator.

Equal ingredient-reference/unit pairs aggregate in original request order. Numeric quantities sum and an all-null group stays null. Category/icon each use three-state input: omitted preserves/inference behavior; explicit string or null overrides; the last explicit value in that aggregate group wins. The accepted batch is all-or-nothing across every aggregate pair. `nextSortIndex` reserves one contiguous range in aggregate input order; abandoned retry ranges cannot commit, while later successful requests may leave gaps only when an already-committed allocation is followed by a later unrelated delete. Concurrent numeric adds sum, null adds retain the serialization-current quantity, simultaneous tombstone restores become first restore plus subsequent active add, and the later successful request wins conflicting overrides. D1 commit order is the only winner order; tests assert both possible request orders, never wall-clock call order.

### Cook-Session Ownership And Lifecycle
- Every JSON cook response carries `X-Request-Id` and uses one exact envelope. Success is `{ ok: true, requestId, data }`. Error is `{ ok: false, requestId, error: { code, message, status }, data? }`; `data` is present as `{ snapshot }` on stale attempt/revision conflicts when a current attempt exists. An inbound trimmed `X-Request-Id` matching `^[A-Za-z0-9._:-]{1,128}$` is echoed; otherwise generate `req_<UUID-v4>`. The response header and envelope value are identical. Every non-101 response sets `Cache-Control: private, no-store` and `Pragma: no-cache`.
- `POST /api/cook-sessions` accepts `{ recipeId, mutationId, adoption? }`, where mutation ids are UUID v4 strings. A newly created attempt returns `201`; active resume or exact replay while the recorded attempt remains current returns `200`; data is `{ snapshot, adoptionOutcome }`. An exact replay returns the recorded adoption outcome together with the recorded attempt's current snapshot, including progress or terminal changes made after the original start response, never the original response snapshot. `adoption`, when present, is `{ version: 1, fingerprint, activeStepIndex, scaleFactor, checkedIngredientIds, checkedStepOutputIds }` and is validated against the server-built current recipe snapshot. Start requests have a separate DO idempotency ledger: unexpired records are retained for 24 hours and replay the recorded attempt/outcome while current, or return `410 attempt_replaced`/`410 attempt_purged` from non-content tombstones after replacement/purge; reusing an id with another payload returns `409 mutation_conflict`; at 64 unexpired unique records the DO returns `429 mutation_ledger_full` rather than evicting a replay guarantee.
- The Worker derives the canonical DO address as `idFromName("${userId}:${recipeId}")`. That deterministic per-user/per-recipe object, not D1, arbitrates start/resume and guarantees one active attempt. Parallel starts serialize in the same object.
- Every new attempt gets a random `attemptId` UUID stored inside the DO and returned in snapshots. D1 stores one history/index row per attempt and retains nullable unique `activeKey = "${userId}:${recipeId}"` as a projection integrity check, but start/resume never consults D1 to decide canonical state.
- Active recipes are cookable by any authenticated user because the current product has public active recipes. Creating any new attempt, including terminal-to-new restart, uses the same `deletedAt: null` visibility rule as recipe detail; missing, soft-deleted, or future visibility-inaccessible recipes all return indistinguishable `404 not_found`, and the terminal/current coordinator state is retained unchanged. No recipe-ownership restriction is introduced. Once created, the pinned attempt remains GET/PATCH/live/complete/abandon/purge accessible to its owner without re-reading Recipe visibility, and active start/resume returns it, even if the recipe is later edited, soft-deleted, or visibility-inaccessible while its row remains; uncoordinated hard deletion is outside this guarantee. A soft-deleted recipe cannot create another attempt. The recipe route falls back to the authenticated caller's own current pinned snapshot when public recipe lookup fails, rendering cook-only content; anonymous callers and authenticated callers without their own attempt receive the ordinary 404.
- `GET /api/cook-sessions/recipes/:recipeId` returns `200 { snapshot }` from the authenticated user's deterministic DO. `PATCH` accepts `{ attemptId, baseRevision, mutationId, operations }`, with one to 64 operations from exactly: `{ type: "set_active_step", stepIndex }`, `{ type: "set_scale_factor", scaleFactor }`, `{ type: "set_ingredient_checked", ingredientId, checked }`, and `{ type: "set_step_output_checked", stepOutputId, checked }`. Accepted and replayed mutations return `200 { snapshot }`.
- Idempotency equality uses canonical validated payloads, never raw bytes: reject unknown fields, normalize adoption checked arrays into pinned order, recursively sort object keys, preserve operation-array order, then hash canonical JSON; request-id/header differences are excluded. PATCH validates every operation before applying any. Operations execute sequentially in array order; duplicate targets are allowed and the last operation wins. One invalid operation rejects the whole request with no state, ledger, projection, revision, or timestamp change.
- Revision-zero defaults are `activeStepIndex: 0`, `scaleFactor: 1`, empty checked arrays, ACTIVE status, null terminal timestamps, and one clock reading used for equal `startedAt`/`updatedAt`. Each accepted non-replayed PATCH increments revision exactly once regardless of operation count or whether final values equal prior values; `updatedAt` becomes `max(clockNow, previousUpdatedAt + 1ms)`. `POST .../complete` and `POST .../abandon` accept `{ attemptId, baseRevision, mutationId }`, increment revision once on the first accepted transition, update time by the same monotonic rule, return `200 { snapshot }`, and enqueue an ordered D1 projection clearing `activeKey`. After replay/id-hash and attempt-id checks, terminal-state precedence comes before base-revision comparison: requesting the opposite terminal target returns `409 terminal_conflict` even when stale; repeating the same target with a fresh id ignores stale base and, when ledger capacity remains, records the no-op response then returns the current snapshot without revision/time change; at capacity it returns `429 mutation_ledger_full`. An ACTIVE attempt with stale base returns `409 stale_revision` before capacity. Completion is private history only and never creates a `RecipeSpoon` or notification.
- The progress/terminal mutation ledger is attempt-scoped, retains up to 256 unexpired UUID-v4 records for 24 hours, and never evicts an unexpired replay guarantee. Replay lookup and request-hash comparison happen first. Only accepted mutations enter the ledger: validation, stale attempt/revision, terminal, capacity, and other rejected requests leave no record. While the attempt remains current, an exact accepted replay returns its recorded status/snapshot without incrementing revision; reuse with another payload returns `409 mutation_conflict`. Replacement atomically converts every unexpired progress/terminal record for the old attempt into a non-content `attempt_replaced` tombstone; purge converts the current attempt's records into `attempt_purged` tombstones. These request-hash/id/attempt/outcome/expiry tombstones remain in coordinator storage for the remainder of 24 hours and return the matching 410 rather than a snapshot. Their natural maximum is 64 start-ledger attempts times 256 records; they are never counted as attempt-bearing content. A new mutation at current-ledger capacity returns `429 mutation_ledger_full`. A received stale revision includes latest snapshot; the client rebases with a fresh UUID/latest revision. After network uncertainty, it first retries the canonical-equivalent original request/id. Unrelated fields merge and reconnecting same-field operation wins when accepted.
- A fresh start mutation while the current attempt is `COMPLETED` or `ABANDONED` atomically creates a new ACTIVE attempt at revision zero and returns `201` with that new snapshot and `server_won` for any supplied adoption. The transaction persists new current/coordinator/start-ledger state, converts every unexpired start record targeting the terminal attempt to a non-content `attempt_replaced` tombstone, and appends the new create projection without discarding older projections; only afterward are sockets attached to the terminal attempt closed with `4000 Attempt replaced`. Old start replays return `410 attempt_replaced` and never resolve to the newer attempt. A start while current is ACTIVE resumes it with `200`.
- `GET /api/cook-sessions/recipes/:recipeId/live` requires a same-origin WebSocket upgrade. Server frames are exactly `{ type: "snapshot", snapshot }` and `{ type: "error", error: { code, message }, snapshot? }`. The server sends an initial snapshot and broadcasts after accepted progress or terminal transitions. Error frames occur only after an accepted upgrade: replacement/wake validation sends `stale_attempt` plus latest snapshot when one exists, then closes `4000`; a purge race sends `session_purging` plus PURGING snapshot, then closes `4001`; missing state sends `not_found` without snapshot, then closes `4000`; state-load failure sends `internal_error` without snapshot, then closes `1011 Internal error`. Send is awaited before close. Ordinary replacement uses that stale error/close sequence; ordinary purge first broadcasts the PURGING snapshot and then closes without a second error frame. Client text/binary data closes immediately with `1003 Read-only subscription` and no error frame; normal client close uses `1000`. Hibernation attachments are `{ version: 1, userId, recipeId, attemptId }`.
- Upgrade attachment failure returns ordinary `500 internal_error` before 101. After 101, initial-snapshot send failure attempts close `1011 Initial snapshot failed`; close failure is swallowed and the socket is left for later event validation. For every error frame, attempt the specified close code/reason even if frame send fails; send/close errors never change DO state or an already-selected HTTP/upgrade outcome. Replacement attempts stale frame then 4000 for every old socket independently; failures do not roll back the new attempt. Accepted non-purge progress/terminal state and projection enqueue commit before fanout; healthy sockets continue, failed send attempts `1011 Broadcast failed`, failed close is swallowed, HTTP remains 200, and no broadcast retry alarm is scheduled. Purge persists fence/barrier/alarm first, then for every socket attempts PURGING snapshot and 4001 independently regardless of send success; failures never roll back, stop projection alarm processing, or change DELETE 202. A surviving old/purging socket is revalidated and closed on its next event/wake. Clients recover by reconnect initial snapshot/GET.
- `DELETE /api/cook-sessions/recipes/:recipeId` accepts `{ attemptId }` and is not mutation-ledger-backed. Precedence is current state first: while any current attempt exists, only its matching id may purge (or continue PURGING); every other id returns `409 stale_attempt` even if it equals `lastPurgedAttemptId`. Only when `currentAttemptId` is null does a body id equal to `lastPurgedAttemptId` return bodyless `204`; another id returns `404`. A matching purge returns `202 { status: "purging", attemptId }` until the coordinator tombstone commits; repeated pending requests also return `202`, then matching/already-purged requests return `204`. A valid purge atomically persists `PURGING`, appends a purge barrier after every older projection, broadcasts that terminal state, and closes existing sockets. While PURGING, start/resume, snapshot, patch, complete, abandon, and new upgrades return `409 session_purging` and append nothing. The alarm drains all earlier projections FIFO, idempotently deletes the current D1 attempt row at the barrier, then atomically removes pinned recipe, progress, current attempt, response-bearing mutation records, projection queue/retry state, and sockets while converting unexpired start/progress/terminal mutation records to non-content `attempt_purged` tombstones. Retained coordinator metadata is only `everStarted`, `lastPurgedAttemptId`, and those 24-hour replay tombstones. They can never recreate/adopt state. A fresh mutation id may start a new empty attempt after purge, but `everStarted` makes adoption resolve `server_won`.
- Failure behavior is fixed: local persistence failure before the PURGING fence leaves state unchanged and returns `500`; broadcast/socket-close failure after persistence never rolls back the fence; older-projection D1 failure retains queue/barrier/content; current-row D1 delete failure retains barrier/content; a crash after D1 delete retries that delete idempotently; failed atomic attempt-key deletion/tombstone write retains content/barrier and retries; success is reported only after the tombstone is committed and attempt-bearing keys are absent.
- `GET /api/cook-sessions/recipes/:recipeId/residue` is an owner-authenticated, same-origin, non-OpenAPI diagnostic used by cleanup. It returns `200 { everStarted, currentAttemptId, lastPurgedAttemptId, attemptBearingKeyCount, pendingProjectionCount, liveSocketCount }` and never returns recipe/progress content. After purge, the exact expected state is `{ everStarted: true, currentAttemptId: null, lastPurgedAttemptId: <purged id>, attemptBearingKeyCount: 0, pendingProjectionCount: 0, liveSocketCount: 0 }` before disposable-user deletion.
- Request precedence is exact. First resolve known route and allowed method: unknown path is 404 without Allow; wrong method is 405 with Allow before auth/body work. For an allowed request, authenticate first. State-changing requests, `/residue`, and upgrades then validate Origin before reading JSON; ordinary snapshot GET does not require Origin. Parse JSON next, then validate/canonicalize the entire route/body, UUIDs, adoption, and all operations before DO state checks, so invalid operation beats stale attempt and invalid Origin beats malformed JSON. In the DO, look up the mutation id first: exact live replay, hash conflict, or replaced/purged tombstone wins. Then require current attempt identity; check PURGING; apply terminal-state rules; compare revision where applicable; perform any required new-attempt Recipe visibility check; and only then check ledger capacity before mutation. Thus opposite terminal target beats ledger-full, same terminal target at capacity returns ledger-full, stale revision beats ledger-full, and PATCH against terminal returns session_terminal before capacity. Unexpected storage exceptions alone map to 500.
- State-changing HTTP requests and upgrades enforce same-origin `Origin`; all endpoints use cookie sessions and are intentionally excluded from public v1/OpenAPI. Status/code/message mapping is exact: `400 invalid_json` / `Request body must be valid JSON.`, `401 authentication_required` / `Authentication required.`, `403 invalid_origin` / `Origin must match request origin.`, `404 not_found` / `Cook session or recipe not found.`, `405 method_not_allowed` / `Method not allowed.`, `409 stale_attempt` / `The cook attempt is no longer current.`, `409 stale_revision` / `Cook session revision is stale.`, `409 mutation_conflict` / `mutationId was already used for a different request.`, `409 terminal_conflict` / `Cook session already has a different terminal status.`, `409 session_terminal` / `Cook session is terminal.`, `409 session_purging` / `Cook session purge is in progress.`, `410 attempt_replaced` / `The recorded cook attempt was replaced.`, `410 attempt_purged` / `The recorded cook attempt was purged.`, `422 validation_error` / `Request body is invalid.`, `429 mutation_ledger_full` / `Cook session mutation ledger is full; retry after records expire.`, and `500 internal_error` / `Internal server error.` Upgrade auth/origin failures are ordinary non-101 JSON responses. `405` sets the route's exact `Allow`; `429` sets `Retry-After` to the ceiling seconds until the oldest ledger record expires.
- Exact `Allow` values/order are: start base `POST`; recipe base `GET, PATCH, DELETE`; `/live` `GET`; `/complete` `POST`; `/abandon` `POST`; `/residue` `GET`. HEAD is not promoted to GET. Unknown paths return 404 without Allow.

### Cook Snapshot, Offline, And Projection
- `CookRecipeSnapshot` is exactly `{ version: 1, id: string, title: string, coverImageUrl: string|null, steps: Step[] }`. Each step is `{ id: string, stepNum: positive integer, stepTitle: string|null, description: string, duration: nonnegative integer|null, ingredients: Ingredient[], stepOutputUses: StepOutputUse[] }`. Each ingredient is `{ id: string, ingredientRefId: string, name: string, quantity: finite number, unit: { id: string, name: string } }`. Each step-output use is `{ id: string, outputStepId: string, outputStepNum: positive integer, outputStepTitle: string|null }`. IDs and title/name strings must be nonempty; empty description is allowed.
- Snapshot construction sorts steps by `(stepNum, id)`, ingredients within a step by `id`, and step-output uses by `(outputStepNum, id)`. Canonicalization recursively sorts object keys lexicographically; preserves array order, stored Unicode code points, case, and whitespace; includes every schema field and emits nullable values as JSON `null`; never emits `undefined`; rejects non-finite numbers; normalizes `-0` to `0`; and otherwise uses ECMAScript JSON number serialization. The fingerprint is lowercase hexadecimal SHA-256 of the UTF-8 canonical recipe-snapshot JSON. Database timestamps and other non-schema fields are excluded.
- `CookSessionSnapshot` is exactly `{ version: 1, attemptId: UUID-v4 string, recipeId: string, recipeFingerprint: 64-char lowercase hex, recipe: CookRecipeSnapshot, revision: nonnegative integer, status, activeStepIndex: integer, scaleFactor: finite number, checkedIngredientIds: string[], checkedStepOutputIds: string[], startedAt: ISO string, updatedAt: ISO string, completedAt: ISO string|null, abandonedAt: ISO string|null }`. Status is `ACTIVE|COMPLETED|ABANDONED|PURGING`; active-step index is zero for an empty recipe and otherwise `0..steps.length-1`; scale is `0.25..50`; checked arrays are duplicate-free subsets of the pinned IDs. ACTIVE has null terminal timestamps; COMPLETED has only `completedAt`; ABANDONED has only `abandonedAt`; PURGING preserves the pre-purge terminal-timestamp shape. Adoption has these same index/scale/subset/uniqueness rules; client checked-array order is ignored and canonicalized to pinned traversal order, while duplicate or unknown ids return `422 validation_error`.
- The DO key already encodes authenticated owner/recipe identity; storage repeats and verifies immutable owner/recipe ids. Coordinator metadata stores `everStarted`, current/last-purged attempt ids, and start replay records. Attempt-bearing storage holds the pinned recipe snapshot/fingerprint, revision, active step, scale factor bounded to `0.25..50`, checked stable ids, status, timestamps, the 256-entry progress ledger, and an ordered pending D1 projection queue; purge removes this entire attempt-bearing set.
- Active cooking renders the pinned snapshot. When the current recipe fingerprint differs, the UI offers `Continue original` or `Start fresh`; start-fresh abandons the old attempt before creating a new one.
- Authenticated local state uses one key `spoonjoy-auth-cook:v1:<userId>:<recipeId>` with `{ version: 1, userId, recipeId, attemptId: UUID-v4|null, snapshot: CookSessionSnapshot|null, pending, inFlight, command }`. `pending` entries are `{ localId: UUID-v4, operation }`; `inFlight` is null or `{ mutationId: UUID-v4, baseRevision: integer, localIds: UUID-v4[], operations }`; `command` is null or `{ type: "start", mutationId, recipeId, adoption }` or `{ type: "complete"|"abandon", mutationId, attemptId, baseRevision }`. `command` and `inFlight` are mutually exclusive. Start commands require empty progress queues; complete/abandon waits until every progress batch is accepted, then persists its command. Start fresh persists/sends abandon, persists its terminal response, then persists/sends start. A record is read only when embedded/key identity matches; logout/user switch removes it.
- Coalesce pending-only same targets by remove-old/append-new. Pending max is 256; overflow rejects visibly and drops nothing. With no inFlight/command, move the first at most 64 pending entries into one inFlight PATCH preserving order, assign one id/base, persist the whole record, then send exactly that PATCH. Persist every start/complete/abandon command before sending it. A storage failure rejects the initiating action and sends nothing. While command exists, progress and other terminal/start controls are disabled. Accepted responses persist canonical snapshot, clear the matching inFlight/command, and only then allow the next transition. No authenticated mutation is sent from memory-only state.
- If persisting any accepted PATCH/start/terminal response fails, render its canonical server snapshot; for PATCH, reapply only still-pending operations, never the accepted inFlight. Reject further actions and network sends. The old disk inFlight/command remains replay-safe. In the same mount, local persistence retries only from the visible `Retry saving` action or one attempt per window `focus` event, never a timer and never another network request. After reload, bootstrap replays the persisted id to recover the response. Sync unblocks only after the accepted state persists. Anonymous progress clears only after both a 200/201 start response and its authenticated record persist; response-write failure retains it.
- Every authenticated mount has a canonical bootstrap barrier. Parse disk but render none of its state/operations and disable controls. If a command exists, replay that exact command first. Otherwise GET canonical state; a same-attempt GET persists its latest base, then resolves any inFlight by replaying its original id; a different attempt persists only the new snapshot with empty queues; 404 clears to idle; 401 clears auth state. Network/408/425/429/5xx retain hidden disk state and retry on `online`, WebSocket reconnect, or explicit Retry, with 429 blocked until Retry-After; other HTTP/protocol errors stay blocked until explicit retry/reload. Persistence failure keeps the barrier closed. A bootstrapped tab that later goes offline may persist local actions, but an offline reload never renders or sends unverified state.
- Every accepted HTTP/WebSocket snapshot transition also passes through the persistence gate. The persisted `snapshot` field is always the canonical server base; queued optimistic operations are derived over it for rendering and are never folded into that field. For a same-attempt higher revision, persist the new base with unchanged still-applicable queues before allowing another send. A different attempt immediately invalidates every old pending/inFlight operation: render only the new canonical snapshot and attempt to persist it with empty queues. If that replacement write fails, the old disk record remains but its operations are quarantined in memory and are never rendered or sent; sync stays blocked with the device error. Reload re-enters the bootstrap barrier rather than trusting that disk record. Persistence success unblocks sync. Lower and equal-identical snapshots perform no write.
- Anonymous storage remains at `spoonjoy-cook-progress:<recipeId>`. New records are v2 `{ version: 2, recipeFingerprint, activeStepIndex, scaleFactor, checkedIngredientIds, checkedStepOutputIds, updatedAt }`; only duplicate-free/bounds-valid v2 with the current fingerprint is adoption-eligible. Existing v1 records parse and continue local-only with their current bounds filtering but are never upgraded or adopted because they cannot prove a fingerprint; anonymous writes keep such a record v1. On authentication, v1 or fingerprint-mismatched v2 sends no adoption (`not_requested`). Any anonymous record clears only after a successful `200` or `201` start response and successful persistence of the resulting authenticated record, regardless of adoption outcome; network/HTTP/local-write errors preserve it for retry. A new anonymous session without legacy state writes v2; mismatched v2 resets to empty current-fingerprint v2 before the next local action.
- WebSocket/HTTP server snapshots with the same attempt and higher revision replace the server base after successful persistence, after which queued operations are reapplied optimistically; lower revisions are ignored. Equal revision plus byte-equivalent canonical snapshot is ignored; equal revision with different content is a protocol error that performs GET and makes no local winner choice. A different server attempt clears old queues/command and adopts through the persistence gate. Error disposition is exact: stale revision persists a fresh id/latest base on the same inFlight; PATCH mutation conflict GETs and retries only the same ACTIVE attempt; start mutation conflict GETs then persists/retries the unchanged start/adoption with a fresh id; terminal mutation conflict retries only the same ACTIVE attempt, otherwise discarding command and adopting fetched state. Stale/replaced clears then GETs; purged/purging/not-found clears to idle/purging; terminal conflicts GET/persist terminal and clear queues/command; 401 clears auth. Network/408/425/429/5xx retain the byte-equivalent persisted inFlight/command and retry only on bootstrap/online/WebSocket-reconnect/explicit Retry, honoring Retry-After. Contract 4xx/unknown responses retain state and block until explicit retry/reload. Recovery writes follow the persistence gate. Adoption never trusts client time; only a pristine object adopts matching anonymous v2, and anonymous clears only after authenticated persistence.
- The DO is canonical. D1 contains owner-private active/history metadata needed for My Kitchen and may temporarily show a stale card. Activating a Continue Cooking card first GETs the DO: ACTIVE navigates; terminal/404 hides the card locally, leaves navigation in place, and calls route revalidation after the GET has ensured any pending alarm is scheduled; network/error keeps the card and shows retry. The same mount never re-adds a locally invalidated card, while a later loader loses it after projection repair. Create/update/complete/abandon projections are processed in attempt order, idempotently, so a terminal old attempt clears its active key before a new attempt claims it. On a failed queue head, the alarm increments persisted retry count from zero to `n` and schedules `now + min(2^n seconds, 60 seconds)`, yielding 2, 4, 8, 16, 32, then 60 seconds forever until success. A successful head resets retry count to zero before draining the next item. Starting a new attempt never discards an older pending terminal projection.
- A live-smoke cleanup pass sets `deadline = now + 90s` and a total budget of 20 cook-session HTTP requests, including discovery, DELETE, and post-delete residue. It first calls owner `/residue` with known user/recipe identity, so a committed start with a lost/malformed response is discoverable. A valid currentAttemptId becomes the DELETE target. Null current id plus zero attempt-bearing/projection/socket counts skips DELETE and proceeds to D1/user proof; null plus any nonzero count is fatal. Before every request, fail if expired; count it; abort after `min(10s, deadline-now)`. Network/abort, 408, 425, 429, and 5xx retry; DELETE 202 also retries. Parse Retry-After only as nonnegative integer delta-seconds; delay is `max(parsedMsOrZero, min(500*2^(requestCount-1),5000))`; fail before sleep at request 20 or when `now+delay >= deadline`. DELETE 409 retries only for exact `stale_attempt` with valid latest snapshot id, switching target without resetting counters. Every other 3xx/4xx/unexpected 2xx/malformed body is fatal. Bodyless DELETE 204 must be followed by exact residue with null current and all zero counts. Then require environment-safe remote D1 attempt-row zero and successful disposable-user deletion; proof/inspection/delete failure fails the pass.
- Cleanup runs in `finally` even when functional smoke assertions fail. If the primary pass fails, record its full evidence, wait exactly 5 seconds, then run one fresh recovery pass with a new request/deadline budget and the latest known attempt target; the recovery repeats DELETE even after an earlier 204 so it can retry proof/deletion. Report `residueClean` separately from process success. Exit zero only when functional smoke passes and primary cleanup succeeds. Functional failure with clean primary cleanup exits nonzero with `residueClean: true`; primary failure plus recovery success deletes the user but exits nonzero with `residueClean: true, cleanupRecovered: true`; two failed passes exit nonzero with `residueClean: false, cleanupIncomplete: true` and leave the user. QA/production gates rerun until smoke exits zero on primary cleanup, and completion is forbidden until attempt-bearing DO count, D1 rows, and disposable user data are all zero. Any number of still-unexpired bounded non-content replay tombstones may remain in the DO and do not count as user content.

### CookSessionIndex Projection Schema
- D1 model `CookSessionIndex` has: `attemptId String @id`, `userId String`, `recipeId String`, nullable unique `activeKey String?`, `status String`, `recipeFingerprint String`, `recipeTitle String`, `recipeCoverImageUrl String?`, `activeStepIndex Int`, `scaleFactor Float`, `checkedIngredientCount Int`, `checkedStepOutputCount Int`, `revision Int`, `projectionSeq Int`, `lastProjectionId String @unique`, `startedAt DateTime`, `updatedAt DateTime`, `completedAt DateTime?`, and `abandonedAt DateTime?`; user/recipe foreign keys cascade on hard delete. Indexes are `(userId,status,updatedAt)`, `(userId,recipeId,startedAt)`, and `(recipeId)` plus the unique active key/projection id constraints. It stores no full recipe/progress/checklist content.
- Queue entries are `{ version: 1, projectionId: "<attemptId>:<sequence>:<UPSERT|DELETE>", attemptId, sequence, kind, payload }`. Initial create sequence is 0; accepted progress/terminal entries use the resulting revision; purge DELETE uses current revision + 1. UPSERT applies only when `sequence >= projectionSeq`, sets `activeKey` only for ACTIVE, and records `lastProjectionId`; exact/older replay is a no-op. DELETE removes only that attempt row when stored sequence is not newer; missing-row replay is a no-op. The per-DO FIFO queue orders old terminal, new active, and purge entries across attempts.
- Every start/progress/terminal/purge transition that appends a projection calls top-level SQLite DO storage writes plus `setAlarm(Date.now())` inside the same explicit `ctx.storage.transaction`; no external I/O occurs inside. Alarm and state/ledger/revision/queue commit atomically. `setAlarm` failure aborts everything, records no mutation id, and returns `500 internal_error`, so client retry is fresh rather than a replay of accepted state. The first alarm is immediate. On D1 head failure, the alarm transaction persists retry count and its next scheduled alarm together; if that transaction or setAlarm fails, the handler throws for Cloudflare's at-least-once retry. A successful retry transaction restores the planned indefinite 2/4/8/16/32/60 schedule. Card GET rearming is additional repair, not the only creation path.

### SavedRecipe

Rerunning is harmless.

Removing membership never unsaves; explicit unsave never removes cookbook membership. Re-save is idempotent. Soft-deleted recipes remain referenced but are hidden until restored.

### RecipeTag
- `RecipeTag(id, recipeId, label, normalizedLabel, kind, source, createdById, createdAt, updatedAt)` stores accepted canonical tags only.
- `kind` is `COURSE` or `CUSTOM`; `source` is `MANUAL` in this release. Only the recipe owner may mutate tags. Manual tags are canonical immediately and have no acceptance transition.
- Course is optional and singular; allowed normalized/storage values are `main`, `side`, `appetizer`, and `dessert`. Custom display labels trim ends, collapse every Unicode whitespace run to one ASCII space, preserve the first accepted display casing while the row exists, and must contain 1..40 Unicode code points after collapse. `normalizedLabel` is `label.normalize("NFKC").toLowerCase()`; uniqueness uses that value. Duplicate add keeps existing casing; hard delete then re-add uses new casing. Course words are reserved. At most 10 custom tags are enforced by a SQLite BEFORE INSERT trigger that ignores normalized duplicates; a partial unique index enforces one COURSE; AFTER INSERT/DELETE/UPDATE triggers bump parent Recipe `updatedAt`, so no-op conflict-ignore/delete-missing calls do not. Ordinary prevalidated over-limit is HTTP 400; a race that loses at the trigger maps to `409 tag_conflict` / `Tags changed; review and retry.` with submitted form values.
- All tag writes use current-state declarative SQL inside one non-interactive transaction. Full replacement deletes absent custom/course rows first, inserts each desired normalized row with conflict-ignore to preserve the row/casing that exists at execution time, and then applies the recipe update. D1 commit order is the serialization order: simultaneous course replacements are later-commit-wins; full replace committed later replaces the whole set; a later delta add/remove applies to that replacement; two adds observed at nine yield one ten-tag success and one tag_conflict. Tests force both commit orders and concurrent delete/re-add casing behavior.
- Accepted course/custom labels participate in public recipe search fingerprints and documents. `search.server.ts` adds RecipeTag count/latest/content hash and sorted canonical tags. Tag mutation never reindexes synchronously. Search rebuilds serialize through runtime table `SearchRebuildLease(current, lease UUID?, expiresAtMs?)`: acquire immediately, then wait exactly 25/50/100/200/400/800/1600 ms before attempts two through eight; a lease expires after 60 seconds. The holder may perform at most three fingerprint/build/atomic-replacement cycles. Delete/all inserts/metadata commit in one non-interactive D1 transaction guarded by its lease; afterward, while still holding the lease, recompute source fingerprint. A mismatch retries from a fresh build; three mismatches, lease exhaustion, or statement failure releases best-effort and returns internal error without querying. Crash leaves the old complete index or one complete committed index and the lease expires. Other searches never query while a live lease exists. A matching postcheck is the response's consistency point; release then query, while a source commit after that point belongs to the next request. Logged-in no-store projections are immediate; anonymous recipe/search responses retain `public, max-age=60, stale-while-revalidate=300`. Future AI proposals require a separate acceptance model.
- Discovery filters use one optional `course=<controlled-value>` and repeatable `tag=<custom-label>` query parameters. Empty tag parameters are ignored; custom filters use the same normalization, deduplicate by normalized label, and cap at 10. Multiple course parameters, invalid course/custom values, or an over-limit tag set produce an HTTP 400 loader response. Search text, owner/visibility scope, course, and tags combine with AND; every distinct custom tag must match (AND, not OR). Unfiltered behavior and ordering stay unchanged.
- Recipe create/edit forms submit exactly one `course` field (`""` means null; missing/repeated/invalid is validation error) and repeated `customTag` fields. Blank custom rows are omitted; first occurrence wins display casing; normalized duplicates collapse; parsed canonical tags sort by normalized label then submitted index. Parse/validate all fields before writes. Create uses one nested Recipe create with nested canonical tag creates. Edit computes a full desired replacement: retain existing ids/casing for normalized labels still present, delete absent rows, create genuinely new rows with submitted casing, and replace course; recipe update plus tag delete/update/create operations execute in one non-interactive Prisma `$transaction([...])`. Any failure rolls back recipe and every tag change; validation returns HTTP 400 with submitted course/custom rows and no write.
