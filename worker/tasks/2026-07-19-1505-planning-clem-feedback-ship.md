## Goal

## Scope

### In Scope
- Make authenticated cook progress canonical in one SQLite-backed Cloudflare Durable Object per user and recipe, with atomic start/resume, same-owner cross-device discovery, and live synchronization.
- Keep anonymous cook progress local-only and prevent authenticated code from reading or uploading anonymous state.
- Pin active cook sessions to the recipe snapshot they started from; support revision-checked progress updates, idempotent completion/abandonment, recipe-edit conflict choices, private resume discovery, projection retry, retention, and owner-authorized purge.
- Store only a complete private registry/discovery projection for unpurged cook objects in D1; never store step, checklist, scale, or recipe-snapshot progress there.
- Route web, REST v1, MCP, and legacy shared shopping additions through one add/restore behavior: checked or deleted items return unchecked at the fresh end position, while active unchecked items preserve position and accumulate quantity.
- Add canonical private `SavedRecipe` state distinct from cookbook membership and social activity, with one-time migration of existing cookbook-derived saved expectations, owner-only paginated search, web controls, and REST v1 endpoints.
- Add recipe-owned manual tags: one controlled course and up to ten custom labels, authoring/editing, bounded filters on My Recipes/search, public read metadata, and REST/MCP read parity.
- Add optional read-time recipe scaling to REST detail and MCP `get_recipe` through one pure shared helper without changing stored quantities or existing field types.
- Add one forward-compatible numeric D1 migration for the new data models, one-time save backfill, and shopping identity repair/index; keep the existing Prisma migration mirror and unrelated migration tooling unchanged.
- Add a dedicated official Workers Vitest project and CI command for the DO/Worker paths, upgrading Vitest to `^4.1.0` and locking a compatible Workers pool, with Istanbul coverage and serialized shared-storage WebSocket execution.
- Add the `CookSession` namespace through the supported legacy `migrations` array using `new_sqlite_classes`. Ship a forward-compatible inert namespace/test/deploy bootstrap in its own reviewed PR with one atomic production deploy; ship the first product protocol through a second atomic activation; after product smoke, restore the existing version-upload/canary/promotion path in a workflow-only PR whose old and new Workers both speak protocol v1.
- Remove existing Pebble-specific runtime categorization and fixtures while preserving neutral API-provider behavior.
- Preserve current navigation unless implementation reveals a concrete accessibility defect; validate every changed surface on mobile and desktop.
- Ship through the repository's existing CI, QA, pull-request, merge, and automatic production workflow, with exact deployed-revision health checks and disposable-data cleanup.

### Out of Scope
- First-party import/upload UI; existing agent/API import remains supported.
- AI tag suggestions, inference jobs, confidence/provenance storage, or proposal-review UI.
- Pebble-specific fields, routes, names, telemetry categories, documentation, or behavior.
- Cross-user collaborative cook sessions, public cook history, or automatic `RecipeSpoon` creation.
- A global tag catalog, tag merge administration, cookbook tags, or personalized save state in MCP/public recipe responses.
- Personalized `isSaved` enrichment on recipe-read endpoints, ongoing cookbook-to-save coupling, or a full rewrite of shopping check/update/delete/clear operations.
- SavedRecipe native-sync/tombstone expansion in this release.
- Deleting `prisma/migrations`, replacing Spoonjoy's CI/deploy system, adding a production deployment ledger, sandboxing Wrangler, or implementing Work Suite/Desk/Slugger orchestration inside the product repository.
- A broad navigation redesign or unrelated search/index architecture rewrite.

## Decisions Made
- Authenticated cook state is Durable Object-canonical; anonymous state is browser-local; D1 is metadata-only.
- The SQLite-backed `CookSession` class uses Cloudflare's fully supported legacy lifecycle: explicit top-level and QA bindings plus retained migration `v1_cook_session` with `new_sqlite_classes`. Declarative `exports` is deferred because its permanent prohibition on version upload would remove Spoonjoy's existing canary controls without providing a required capability.
- The bootstrap PR contains exported class `CookSession`, binding `COOK_SESSIONS`, top-level/QA `v1_cook_session` `new_sqlite_classes` migration, `wrangler.workers-test.json`, `vitest.workers.config.ts`, Vitest `4.1.10`, Workers pool `0.18.6`, and a source-controlled one-time atomic production path. It proves the lifecycle boundary before the product PR enables endpoints; failure before boundary verification is repaired forward. The bootstrap Worker intercepts every syntactically valid `/api/cook-sessions` request and returns `503 cook_session_protocol_unavailable`, `Retry-After: 1`, and `Cache-Control: private, no-store`; an attempted WebSocket upgrade remains an HTTP 503. The bootstrap DO returns that same response, without storage mutation, for every recognized future internal request carrying `X-Spoonjoy-Cook-Protocol: 1`, while retaining its exact bootstrap probe. The product Worker adds that header to every DO request, propagates the retryable response, and the client retries it through the frozen 5xx path. The product DO retains the private bootstrap probe so an old Worker/new DO overlap is also valid. Cross-version Workers tests cover both directions. The first product activation is atomic to minimize eventual-consistency overlap and is repaired forward rather than rolled back to the inert Worker. Only after production product smoke does a workflow-only PR restore gradual canaries; at that point both deployed versions speak protocol v1.
- Exact same-origin `/api/cook-sessions/:recipeId/*` HTTP/WebSocket endpoints authenticate in the Worker, validate `Origin` for mutations/upgrades, derive the DO with `idFromName(JSON.stringify([userId, recipeId]))` server-side, and return `private, no-store`; WebSocket upgrade responses bypass response cloning. Session principals retain their existing full first-party access. Bearer list/detail/WebSocket routes require expanded `kitchen:read`; bearer start/PATCH/complete/abandon/restart/purge require expanded `kitchen:write`. Missing scope returns 403 `insufficient_scope` in the cook error envelope, distinct from 403 `origin_forbidden`.
- Private `GET /api/cook-sessions` reads only the authenticated owner's D1 active rows ordered by `updatedAt DESC, recipeId DESC`. The home polls it every five seconds only while its initial list is empty and the page is visible/online, with immediate revalidation on mount, focus, visibility, and online events.
- The recipe-scoped contract is `POST .../start`, `GET/PATCH .../:recipeId`, `POST .../:recipeId/{complete|abandon|restart}`, `GET .../:recipeId/socket`, and owner-only `DELETE .../:recipeId` for purge. The Worker builds start/restart snapshots from D1; clients never submit trusted recipe content.
- Snapshot version 1 pins title, servings, ordered step IDs/numbers/titles/descriptions/durations, ingredient IDs/names/quantities/units, and step-output-use IDs/links plus a deterministic content hash. The DO stores it with `active|completed|abandoned` state.
- The DO mints `attemptId`; every mutation carries `attemptId`, `expectedRevision`, and `mutationId`. Duplicate mutation IDs replay their result, stale attempts/revisions return 409 with the canonical revision/snapshot, simultaneous starts resume one active attempt, and restart creates a new attempt only by atomically abandoning the old one. Recipe mismatch offers `continue original` or `abandon and start current`; no command silently remaps IDs.
- Every state-changing HTTP operation (`start`, PATCH, `complete`, `abandon`, `restart`, purge/cleanup), every WebSocket upgrade, and every alarm enters one per-object in-memory promise mutex owned by `CookSession`; no caller deletes `CookSessionIndex` directly. Each operation re-reads SQLite state after acquisition and alarms/upgrades discard stale pre-lock observations. If an upgrade wins before purge, it accepts and sends its initial snapshot while holding the mutex, then purge observes and closes it; if purge wins, the later upgrade sees absent state and returns HTTP 404 without upgrading. If restart wins before a due purge, it clears terminal/purge time and the later alarm preserves the new active attempt; if purge wins, it removes state and the later restart returns 404. The mutex remains held across the brand-new D1/SQLite handshake and cleanup waits for it, then re-reads local state before acting. Brand-new start mints its attempt, writes the conditional D1 revision-0 row first, then commits SQLite state. D1 failure leaves no DO state and returns 503. SQLite failure returns 503 and conditionally deletes that exact D1 attempt/revision; failed compensation leaves an enumerable row whose detail is 404. After a crash/restart, a start that finds no local state but an exact orphan row conditionally removes it and retries inside the mutex. Only after initial success do canonical state and highest-revision pending projection persist before later D1 writes; later failures retry monotonically without resurrecting terminal rows. One scheduler owns projection retry and 24-hour retention, then closes sockets, deletes alarm/storage, and removes the registry row.
- D1 is the complete registry for every unpurged object. Test/smoke cleanup enumerates owner rows and calls each server-derived DO; the DO mutex either purges canonical state or, after proving local state absent, conditionally removes the exact enumerated orphan row. Live smoke retains its owner session through purge. Remote QA broad cleanup uses its existing privileged D1 channel to insert a random five-minute `kitchen:read kitchen:write` bearer credential for each exact validated disposable owner, storing only token hash/prefix; it calls the normal owner-only purge route with `Authorization: Bearer <token>` and `Origin: <exact target origin>`, deletes the temporary credential in `finally`, verifies zero index rows, and only then deletes user data. Any auth/purge/verification failure refuses user deletion. Production broad cleanup stays read-only; exact production smoke uses its still-live owner session. Partial failures remain retryable and emit bounded privacy-safe errors through existing observability.
- Shopping uses one SQLite `INSERT ... ON CONFLICT ... DO UPDATE ... RETURNING` statement and a partial unique active-identity index over `shoppingListId`, `ingredientRefId`, and `COALESCE(unitId, '')`. Migration survivor is lowest `(sortIndex,id)` under JavaScript-style UTF-16 code-unit ID order; quantity is sum of non-null values or null when all null; checked is false if any unchecked; checkedAt is null when resulting unchecked or the minimum non-null checkedAt when all checked; deletedAt stays null; categoryKey/iconKey use survivor value or the first non-null value in survivor order; survivor updatedAt advances by max(bound migration time, prior +1ms). Extras preserve all data except deletedAt becomes bound migration time and updatedAt advances by the same rule. For active conflict, quantity matrix is `(null,null)->null`, `(null,n)->n`, `(n,null)->n`, `(n,m)->n+m`; incoming non-null category/icon replaces, null preserves. Active unchecked retains ID/position; checked retains ID, clears checked/checkedAt/deletedAt, and moves to `COALESCE(MAX(active sortIndex),-1)+1`; deleted never conflicts and re-add creates a new active ID at that rule with incoming values only. Every success advances updatedAt by max(bound now, prior +1ms). No app retry; failure is atomic and surfaces existing adapter error.
- SavedRecipe key is `(userId, recipeId)` with `savedAt`; backfill deduplicates memberships by `Cookbook.authorId` and sets `savedAt = MAX(RecipeInCookbook.updatedAt)`. Afterward only explicit save/unsave changes it. REST uses `GET /api/v1/saved-recipes`, idempotent `PUT`/`DELETE /api/v1/saved-recipes/:recipeId`, existing kitchen scopes/idempotency, and `Cache-Control: private, no-store` plus credential-varying headers. Lists order by `savedAt DESC, recipeId DESC`; a versioned base64url JSON cursor carries both values and applies the matching strict-less-than predicate, with malformed cursors returning 400.
- The product schema is frozen as follows. Add `User.savedRecipes: SavedRecipe[]`, `User.cookSessionIndexes: CookSessionIndex[]`, `Recipe.savedBy: SavedRecipe[]`, and `Recipe.tags: RecipeTag[]`. `Recipe.course` is nullable `String` with no default, raw `CHECK (course IS NULL OR course IN ('main','side','appetizer','dessert'))`, and modeled index `(course, deletedAt, updatedAt)`. `SavedRecipe` fields are `userId String`, `recipeId String`, and `savedAt DateTime @default(now())`; it has composite primary key `(userId, recipeId)`, `user`/`recipe` relations with cascading deletes, and indexes `(userId, savedAt, recipeId)` plus `(recipeId)`. `RecipeTag` fields are `id String @id @default(cuid())`, `recipeId String`, `label String`, `normalizedLabel String`, `createdAt DateTime @default(now())`, and `updatedAt DateTime @default(now()) @updatedAt`; `recipe` cascades on delete, `(recipeId, normalizedLabel)` is unique, and `(normalizedLabel, recipeId)` is indexed. `CookSessionIndex` fields are `userId String`, `recipeId String`, `attemptId String`, `status String`, `revision Int`, `title String`, `startedAt DateTime`, `updatedAt DateTime`, and `terminalAt DateTime?`, all without defaults; it has composite primary key `(userId, recipeId)`, `user` cascading on delete, deliberately no recipe relation, raw checks `status IN ('active','completed','abandoned')`, `revision >= 0`, and `length(title) <= 200`, and indexes `(userId, status, updatedAt, recipeId)` plus `(status, terminalAt)`. It has no snapshot/progress/checklist/scale columns. Exact index names are `Recipe_course_deletedAt_updatedAt_idx`, `SavedRecipe_userId_savedAt_recipeId_idx`, `SavedRecipe_recipeId_idx`, `RecipeTag_recipeId_normalizedLabel_key`, `RecipeTag_normalizedLabel_recipeId_idx`, `CookSessionIndex_userId_status_updatedAt_recipeId_idx`, `CookSessionIndex_status_terminalAt_idx`, and raw `ShoppingListItem_active_identity_key`; migration 0024 drops only `ShoppingListItem_shoppingListId_unitId_ingredientRefId_key` before creating it.
- Saved REST responses use the existing `{ok, requestId, data}` envelope. `GET /api/v1/saved-recipes?q=&limit=&cursor=` returns `{recipes: Array<RecipeSummary & {savedAt}>, nextCursor}`; searchable fields are recipe title, description, chef username, course, and tag label, compared case-insensitively after trimming/collapsing `q`. The unpadded base64url cursor decodes exactly to `{v:1,savedAt:<ISO string>,recipeId:<string>}`. `PUT .../:recipeId` accepts only `{clientMutationId}`, returns 201 on first save or 200 on an existing save with `{saved:true,recipeId,savedAt,mutation}`; `DELETE` accepts the same body and always returns 200 with `{saved:false,recipeId,mutation}`. Missing/soft-deleted recipes return existing `not_found`; invalid query/cursor/body returns existing `validation_error` with a `field` detail.
- Neutral metadata is top-level `course: string|null` and `tags: string[]` on REST recipe list/detail/search and MCP `search_recipes`/`get_recipe`/`search_spoonjoy`. Before `Recipe.course` enters the generated Prisma shape, MCP import gets an explicit mutation projection so the raw imported recipe can never leak new schema fields. Read-specific wrappers surround existing `recipeDetail`, `formatRecipe`, and `formatRecipeSummary`; existing base output remains byte-shaped for native sync, REST recipe/step/ingredient mutation and recovery responses, MCP create/update/import, and cookbook summaries. OpenAPI keeps existing `RecipeSummary`/`RecipeDetail` for those consumers and adds exact `RecipeReadSummary`/`RecipeReadDetail` schemas for REST list/detail only. Unit 4.1 also owns recipe `SearchDocument` course/tag metadata and a deterministic SHA-256 content hash over recipe tags ordered by recipeId/normalizedLabel/id with fixed keys and normalized timestamps; same-count/same-max-time replacements therefore invalidate freshness before filter UI work. Regression tests enumerate every unchanged consumer and generated example. Web display/filter scope is recipe detail, My Recipes, and global search only. Optional scaling applies only to REST detail and MCP `get_recipe`; when present it adds top-level `scale:{factor,appliedTo:'ingredient_quantities',decimalPlaces:6}` and scales ingredient quantity. OpenAPI adds `scale` query `number` bounds `0.1..100` and optional `scale` response metadata only on `RecipeReadDetail`; generated playground and developer docs show both forms. Absent scale is byte-shape compatible with the new post-metadata unscaled read payload, not the pre-feature payload; invalid scale maps to REST `validation_error` field `scale` and MCP invalid-argument.
- Cook HTTP is frozen to `GET /api/cook-sessions`, `POST /api/cook-sessions/:recipeId/start`, `GET|PATCH|DELETE /api/cook-sessions/:recipeId`, `POST .../:recipeId/{complete|abandon|restart}`, and `GET .../:recipeId/socket`. Success is `{state}` for detail/mutations, `{sessions}` for list, and empty 204 for purge; start returns 201 when created or 200 when resumed, all other state mutations return 200. Errors are `{error:{code,message,retryable,state?}}` with 400 `invalid_request`, 401 `authentication_required`, 403 `insufficient_scope|origin_forbidden`, 404 `not_found`, 409 `stale_attempt|stale_revision|recipe_changed`, and 503 `cook_session_protocol_unavailable|projection_unavailable|purge_incomplete`. Mutation/restart/terminal bodies contain exactly `attemptId`, `expectedRevision`, and `mutationId`; PATCH also contains nonempty `changes` from `activeStepIndex`, `scaleFactor`, `checkedIngredientIds`, and `checkedStepOutputIds`. IDs must exist in the pinned snapshot, arrays are unique, step index is in range, and cook scale preserves the current UI bound `0.25..50`.
- Cook state is `{version:1,recipeId,attemptId,status,revision,snapshotHash,snapshot,progress,startedAt,updatedAt,terminalAt}` with ISO timestamps. Snapshot is `{version:1,recipe:{id,title,servings},steps:[{id,stepNum,stepTitle,description,duration,ingredients:[{id,name,quantity,unit:{id,name}}],usingSteps:[{id,inputStepNum,outputStepNum}]}]}`; nullable source values stay null. Steps sort by numeric stepNum then ID, ingredients by ID, and uses by numeric inputStepNum/outputStepNum then ID; every ID comparison is `(a < b ? -1 : a > b ? 1 : 0)` on JavaScript UTF-16 code units, never locale/SQL collation. `snapshotHash` is lowercase SHA-256 hex of UTF-8 `JSON.stringify` over fixed key order. Progress is `{activeStepIndex,scaleFactor,checkedIngredientIds,checkedStepOutputIds}`. List items are exactly `{recipeId,attemptId,status:'active',revision,title,startedAt,updatedAt}`; snapshot title is full, D1 title truncates to 200 Unicode code points.
- The SQLite DO schema has no defaults and all values are explicit. `session` is `singleton INTEGER PRIMARY KEY NOT NULL CHECK(singleton=1), version INTEGER NOT NULL CHECK(version=1), recipe_id TEXT NOT NULL, attempt_id TEXT NOT NULL, status TEXT NOT NULL CHECK(status IN ('active','completed','abandoned')), revision INTEGER NOT NULL CHECK(revision>=0), snapshot_hash TEXT NOT NULL, snapshot_json TEXT NOT NULL, active_step_index INTEGER NOT NULL CHECK(active_step_index>=0), scale_factor REAL NOT NULL CHECK(scale_factor>=0.25 AND scale_factor<=50), checked_ingredient_ids_json TEXT NOT NULL, checked_step_output_ids_json TEXT NOT NULL, started_at INTEGER NOT NULL CHECK(started_at>=0), updated_at INTEGER NOT NULL CHECK(updated_at>=started_at), terminal_at INTEGER NULL, purge_at INTEGER NULL`, plus `CHECK((status='active' AND terminal_at IS NULL AND purge_at IS NULL) OR (status!='active' AND terminal_at IS NOT NULL AND purge_at=terminal_at+86400000))`. `mutation_receipt` is `attempt_id TEXT NOT NULL, mutation_id TEXT NOT NULL, request_hash TEXT NOT NULL CHECK(length(request_hash)=64), base_revision INTEGER NOT NULL CHECK(base_revision>=0), result_revision INTEGER NOT NULL CHECK(result_revision>=base_revision), response_status INTEGER NOT NULL CHECK(response_status BETWEEN 200 AND 599), response_json TEXT NOT NULL, created_at INTEGER NOT NULL CHECK(created_at>=0), PRIMARY KEY(attempt_id,mutation_id)`; response JSON is the exact mutation `{state}` or error envelope returned with response_status. `pending_projection` is `singleton INTEGER PRIMARY KEY NOT NULL CHECK(singleton=1), revision INTEGER NOT NULL CHECK(revision>=0), payload_json TEXT NOT NULL, attempt_count INTEGER NOT NULL CHECK(attempt_count>=0), next_attempt_at INTEGER NOT NULL CHECK(next_attempt_at>=0)`. Times are Unix ms. Hash canonicalization emits ordered operation/body/changes and sorted unique checked IDs; hash is lowercase SHA-256 UTF-8 JSON. Receipts live through 24-hour terminal retention. Projection retries 1/5/30/120/600s; alarm uses earlier deadline with purge first; purge closes sockets, deletes D1, then alarm/storage; D1 failure preserves storage and retries after 60s.
- WebSockets are receive-only snapshot fan-out. Server frames are exactly `{type:'snapshot',state:<CookState>}` or `{type:'error',error:{code,message,retryable}}`. A successful upgrade sends exactly one current snapshot before returning the upgrade response. Every committed mutation sends exactly one snapshot per attached live socket, with strictly increasing revisions and no duplicate same-revision fan-out; no frame is sent after a socket closes or purge begins. Client data receives error frame `client_messages_unsupported` and then closes 1003/`client-messages-unsupported`; restart closes old attempt 4009/`stale-attempt`; terminal/purge close 1000 with named reasons. Reconciliation is total: clean 1000 stops/list-revalidates; after 4009, a different active attempt is adopted/reopened, the same active attempt stops with surfaced `stale_attempt_reconciliation_failed`, terminal/404 stops, 401/403 stops auth error, 409 adopts canonical active or stops terminal, 5xx/network backs off, other 4xx stops. For 1006/1011/1012/1013, active reconnects after 1/2/5/10s, terminal/404 stops/list-revalidates, 401/403/other 4xx stops, 409 adopts active or stops terminal, 5xx/network retries. Unlisted close codes stop protocol error. Offline pauses; online retries immediately; successful open resets delay. Telemetry remains the frozen allowlist.
- The inert lifecycle bootstrap exposes `POST /.well-known/spoonjoy-cook-session-bootstrap` with an empty body only while `COOK_SESSION_BOOTSTRAP_MODE='1'`. Binding is `COOK_SESSIONS`; the Worker derives `bootstrap:<CF_VERSION_METADATA.id>` and sends `POST https://cook-session.internal/__bootstrap/probe`, header `X-Spoonjoy-Internal-Probe: 1`, body `{\"version\":1}`. The DO accepts that exact probe, creates/inserts/selects/drops `__bootstrap_probe`, calls `deleteAll`, verifies zero user tables, and returns `{ok:true,storage:'sqlite',residue:0}`; Worker adds `workerVersionId` and the normal version header. Separately, recognized future protocol-v1 public/internal cook requests return the frozen retryable 503 contract, while malformed/unrecognized requests return 404. QA/production call the probe twice and assert exact body/version/residue.
