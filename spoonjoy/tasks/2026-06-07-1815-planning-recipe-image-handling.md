# Planning: Recipe Image Handling End to End

**Status**: approved
**Created**: 2026-06-07 18:16

## Goal
Fix Spoonjoy recipe image handling so uploaded recipe and spoon photos are stored upright, accepted formats match product intent, recipe covers show the right empty/generated states, AI redraws run reliably after assignment, and MCP clients can participate in the same image workflow.

## Upstream Work Items
- None

## Scope

### In Scope
- Preserve uploaded JPEG display orientation while stripping private metadata, using a sanitized EXIF Orientation tag rather than Worker-side pixel rotation/transcoding, so mobile photos, including the new risotto upload, render upright after storage in browser/photo surfaces that honor EXIF orientation.
- Restrict recipe and spoon uploads to JPEG, PNG, and WebP in client validation, server validation, hidden file inputs, tests, and user-facing format copy; add missing server validation for browser spoon photo uploads.
- Keep raw uploaded photos visible immediately after save, then schedule background AI stylization for chef-upload recipe covers and origin-cook spoon covers; silently swap to `stylizedImageUrl` when generation succeeds.
- Keep the existing warm editorial cookbook redraw prompt as the stylization prompt and remove the text-only fallback for upload stylization so failed redraws do not invent a different dish.
- Update OpenAI image generation code to match current SDK/API response semantics, persist generated bytes to R2, and keep local/test fallback behavior deterministic.
- Replace corrupted Chef RJ-style recipe no-photo fallback with a real empty-state behavior until an AI placeholder cover finishes; treat the generated AI placeholder as a real cover once present.
- Capture image generation failures, quota skips that matter, and this repair path through the existing PostHog server telemetry/error-tracking path with enough recipe, cover, source, and operation metadata for PostHog alerting; document the PostHog email alert filter because alert delivery is PostHog project configuration rather than Spoonjoy code.
- Add MCP connector support for image uploads and spoon photo uploads so MCP-originated images use the same validation, storage, cover creation, stylization, and PostHog instrumentation paths as browser uploads.
- Repair the known risotto bad data only, after identifying the affected recipe/cover and preserving the current row until the replacement is verified.
- Update relevant docs for upload format, MCP image support, PostHog alerting expectation, and smoke/manual-data hygiene if the primary task adds new smoke flows.

### Out of Scope
- AI stylization for imported recipe covers.
- Building a separate email notification system.
- Changing the approved stylization UX to show user-facing generation errors, blocking spinners, or approval prompts.
- Reworking non-image recipe CRUD, cookbook, shopping-list, auth, or notification behavior except where tests need fixture updates.
- Broad production data cleanup beyond the single known risotto image repair.

## Decisions Made
- GIFs are not wanted; accepted image upload formats are JPEG, PNG, and WebP.
- Only the new risotto image is known to need existing-data repair.
- Raw uploaded photos should appear immediately and silently switch to AI stylized variants when ready.
- Stylization applies now to chef-upload recipe covers and origin-cook spoon photos; imported covers remain later work.
- Do not build a Spoonjoy email notification system for image failures; use existing PostHog server telemetry/error tracking and configure email alerting in PostHog.
- The existing warm editorial cookbook stylization prompt is approved.
- Current OpenAI image docs show GPT image model outputs as base64 image data for the durable path, so Spoonjoy should persist decoded bytes instead of relying on temporary returned URLs.
- MCP image upload contract is Option A: add dedicated upload tools that accept `{ imageBase64, mimeType, filename }`, store through the shared image-storage helper, and return `/photos/...`; recipe and spoon tools continue to accept URLs.

## Context / References
- `app/lib/image-storage.server.ts` validates and stores uploads, currently strips JPEG APP1 metadata before pixel orientation normalization.
- `app/components/recipe/RecipeImageUpload.tsx` and `app/components/recipe/SpoonDialog.tsx` contain client-side upload validation/copy.
- `app/routes/recipes.new.tsx`, `app/routes/recipes.$id.edit.tsx`, and `app/lib/recipe-detail.server.ts` are the browser recipe/spoon upload entry points.
- `app/lib/recipe-cover.server.ts` resolves visible cover URLs and currently returns a Chef RJ-style SVG fallback when no cover URL exists.
- `app/lib/image-gen.server.ts`, `app/lib/ai-placeholder-cover.server.ts`, and `app/lib/spoon-cover-stylization.server.ts` own prompts, OpenAI image calls, quota spending, and background cover updates.
- `app/lib/analytics-server.ts` exposes PostHog `captureException` and controlled server event capture.
- `app/lib/spoonjoy-api.server.ts`, `app/lib/mcp/spoonjoy-tools.server.ts`, and `docs/ouroboros-mcp.md` define and document MCP/API tool behavior; current spoon tools accept `photoUrl` only.
- `prisma/schema.prisma` stores cover variants in `RecipeCover.imageUrl` and `RecipeCover.stylizedImageUrl`.
- Official PostHog docs: `https://posthog.com/docs/error-tracking/alerts` confirms `$exception`-based error tracking can feed alerting, including email via trend alerts.
- Official OpenAI docs/spec: `https://developers.openai.com/api/docs/guides/image-generation` and `https://api.openai.com/v1/images/edits` document base64 image outputs and edit requests that use multipart uploads or JSON `images` references.
