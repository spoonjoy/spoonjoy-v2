## Product And Data Contracts

### Shopping-List State Matrix

| Existing row | Quantity after add | Checked state | Ordering | Category/icon |
| --- | --- | --- | --- | --- |
| No row | Requested quantity exactly, including `null` | Unchecked; `checkedAt` and `deletedAt` null | Newly allocated end position | Per field, explicit string or null stores exactly; omitted infers from ingredient name through `resolveIngredientAffordance` |
| Active, unchecked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Unchecked | Unchanged | Preserve unless request explicitly overrides |
| Active, checked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Reset unchecked; clear `checkedAt` | Move to a newly allocated end position | Preserve unless request explicitly overrides |
| Deleted tombstone | Requested quantity exactly, including `null`; never add stale quantity | Reset unchecked; clear `checkedAt` and `deletedAt` | Move to a newly allocated end position | Per field, explicit string or null stores exactly; omitted preserves stored non-null and infers only stored-null through `resolveIngredientAffordance` |

Recipe batch adds aggregate equal ingredient-reference/unit pairs before mutation: numeric requested quantities sum, while an all-null group remains null. Web, REST, and MCP call the same D1-safe helper.

All shopping-list item writes use an atomic list-level lease/version batch. `ShoppingList` adds `mutationVersion Int @default(0)`, nullable `mutationLease String?`, and `nextSortIndex Int @default(0)`. Before an attempt, one parameterized `INSERT ... ON CONFLICT(userId) DO NOTHING` creates a missing list with a caller-generated nonempty id, version/index zero, null lease, and database timestamps; concurrent losers then read the one winner, and this bootstrap never creates an item or consumes version/index. Every attempt reads that ShoppingList row and observes version `V` first, then reads its items; those reads are sequential and never parallel or items-first. It submits one non-interactive D1 transaction of guarded parameterized raw SQL: conditionally acquire a random lowercase UUID-v4 lease only where `mutationVersion = V AND mutationLease IS NULL`, run every item/create/delete/reorder statement only where that lease exists, allocate the required sort range, then clear the lease and increment version. Because the version read precedes the item read, any concurrent commit visible during or after the item read makes the later version CAS fail; a request can commit only from item state associated with its observed version. If acquisition changes zero rows, every guarded statement is a no-op and the whole request retries from a fresh version-first read. Eight attempts are allowed with fixed delays 5, 10, 20, 40, 80, 160, and 320 ms; exhaustion throws `ShoppingListBusyError`. REST maps it to additive `409 write_conflict` / `Shopping list changed too frequently; retry.`, the web action returns the same 409 message, and MCP returns a retryable tool error with that text. Migration `0024_shopping_list_mutation_coordination.sql` initializes `nextSortIndex` to each list's current `MAX(sortIndex)+1` (zero for empty), version zero, and null lease. Every item mutation uses this coordinator.

Equal ingredient-reference/unit pairs aggregate in original request order. Numeric quantities sum and an all-null group stays null. Category/icon use independent three-state input. Explicit string or null always stores exactly; the last explicit aggregate value per field wins. Omitted on an active row preserves that field even when null. Omitted on a missing row infers that field from canonical ingredient name through `resolveIngredientAffordance`. Omitted on a tombstone preserves a stored non-null field but infers only a stored-null field; it never recomputes a non-null tombstone affordance. Web, REST, MCP, and recipe batches pass the same sentinel semantics. The accepted batch is all-or-nothing. `nextSortIndex` reserves one contiguous range in aggregate input order; an aborted/retried transaction never consumes its proposed range. Successful checked/tombstone moves and deletes may leave vacated sort-index gaps, but no failed attempt creates one. Concurrent numeric adds sum, null adds retain serialization-current quantity, simultaneous tombstone restores become first restore plus subsequent active add, and later commit wins conflicting overrides. D1 commit order is the only winner order; tests assert both request orders.

Nullable-unit identity is database-enforced, not inferred from Prisma's nullable composite unique. Before adding the null-safe index, migration 0024 groups rows by `(shoppingListId, ingredientRefId, COALESCE(unitId,''))`. In each duplicate group it orders active before tombstoned, then `sortIndex`, `createdAt`, `id`; the first row is survivor. If active rows exist, survivor quantity becomes the finite sum of their non-null quantities or null when all are null, survivor is deleted only when none are active, survivor is unchecked if any active row is unchecked otherwise keeps the first active checked timestamp, and category/icon become the first non-null value in that same order; if all rows are tombstones the first row is retained unchanged. Every nonsurvivor is deleted. The transaction then creates exact unique expression index `ShoppingListItem_list_ingredient_unit_nullsafe_key` on `(shoppingListId, ingredientRefId, COALESCE(unitId,''))`; ids are validated nonempty so `''` cannot collide with a real unit id. Prisma keeps its representable composite declaration, while all writers and tests treat the expression index as authoritative.

Three-state inputs are exact before aggregation. REST JSON and MCP: absent/`undefined` means omitted, JSON/tool `null` means explicit null, and any string including `""` means that exact explicit string; other types and unknown fields are validation errors. Web FormData: zero entries means omitted, exactly one string entry including `""` is explicit, a File or repeated field is `400`; the literal text `null` is a string, never null. Recipe ingredient batches have no affordance fields and therefore pass omitted. Free-text item creation also passes omitted after deriving canonical ingredient name. Quantity inputs and every aggregate/result must be finite and within `[-Number.MAX_VALUE, Number.MAX_VALUE]`; overflow maps to each surface's existing 400 validation shape with exact message `Quantity is outside supported numeric range.`

Every Prisma/SQLite shopping integer is bounded `0..2_147_483_647`. Before one accepted batch, require `mutationVersion < 2_147_483_647`, every retained `sortIndex` in range, and `nextSortIndex + reservedCount <= 2_147_483_647`; check with widened SQL arithmetic before writes. Failure changes nothing and throws `ShoppingListCapacityError`. REST returns `409 capacity_exceeded` / `Shopping list numeric capacity reached.`, web returns status 409 with that message, and MCP returns a non-retryable tool error with that text. Lease/contention retries never mask capacity or quantity errors.

### Cook Snapshot, Offline, And Projection
- `CookRecipeSnapshot` is exactly `{ version: 1, id: string, title: string, coverImageUrl: string|null, steps: Step[] }`. Each step is `{ id: string, stepNum: safe integer, stepTitle: string|null, description: string, duration: safe integer|null, ingredients: Ingredient[], stepOutputUses: StepOutputUse[] }`. Each ingredient is `{ id: string, ingredientRefId: string, name: string, quantity: finite number, unit: { id: string, name: string } }`. Each step-output use is `{ id: string, outputStepId: string, outputStepNum: safe integer, outputStepTitle: string|null }`. Stable ids must be nonempty; stored strings including empty are preserved. Zero/negative step/output/duration values are legal; unsafe integers are invalid snapshot source data. activeStepIndex addresses sorted-array position.
- Snapshot construction sorts steps by `(stepNum, id)`, ingredients within a step by `id`, and step-output uses by `(outputStepNum, id)`. Canonicalization recursively sorts object keys lexicographically; preserves array order, stored Unicode code points, case, and whitespace; includes every schema field and emits nullable values as JSON `null`; never emits `undefined`; rejects non-finite numbers; normalizes `-0` to `0`; and otherwise uses ECMAScript JSON number serialization. ### CookSessionIndex Projection Schema
- D1 model `CookSessionIndex` has: `attemptId String @id`, `userId String`, `recipeId String`, nullable unique `activeKey String?`, `status String`, `recipeFingerprint String`, `recipeTitle String`, `recipeCoverImageUrl String?`, `activeStepIndex Int`, `scaleFactor Float`, `checkedIngredientCount Int`, `checkedStepOutputCount Int`, `revision Int`, `projectionSeq Int`, `lastProjectionId String @unique`, `startedAt DateTime`, `updatedAt DateTime`, `completedAt DateTime?`, and `abandonedAt DateTime?`; user/recipe foreign keys cascade on hard delete. - Persisted queue entries are a discriminated union. UPSERT is `{ version: 1, projectionId: "<attemptId>:<sequence>:UPSERT", attemptId, sequence, kind: "UPSERT", payload }`, where payload is exactly `{ userId, recipeId, status: "ACTIVE"|"COMPLETED"|"ABANDONED", recipeFingerprint, recipeTitle, recipeCoverImageUrl: string|null, activeStepIndex, scaleFactor, checkedIngredientCount, checkedStepOutputCount, revision, startedAt, updatedAt, completedAt: string|null, abandonedAt: string|null }`. DELETE is `{ version: 1, projectionId: "<attemptId>:<sequence>:DELETE", attemptId, sequence, kind: "DELETE", payload: { userId, recipeId } }`; PURGING is never UPSERTed. Sequence/revision/count/index fields are nonnegative integers (scale finite in range), fingerprints/ids/status/timestamp nullability match the snapshot, and every timestamp is canonical `Date.toISOString()` text. Initial create sequence/revision is 0; progress/terminal UPSERT uses resulting revision; purge DELETE uses current revision + 1. Enqueue and dequeue validate version, exact fields, projectionId components, immutable owner/recipe/attempt, sequence/payload revision equality for UPSERT, and timestamp/status shape. Any malformed persisted head remains queued and has the exact fail-closed precedence above; `alarm()` throws and it is never dropped or sent to D1. Against an existing D1 row, matching `lastProjectionId` is an exact no-op; greater sequence applies; lower sequence is a no-op; equal sequence with a different projectionId is projection corruption, throws, and never overwrites/deletes. A missing-row UPSERT applies; a missing-row DELETE replay is a no-op. A greater DELETE removes the row; lower is a no-op; equal-different is corruption. Applied UPSERT sets activeKey only for ACTIVE and records lastProjectionId. FIFO orders old terminal, new active, and purge across attempts.
- Projection queue capacity is 1,024 entries with the final slot reserved exclusively for purge DELETE. Start-create, PATCH, first terminal, and terminal-to-new create require current length `< 1,023` before their atomic append; otherwise they return `503 projection_backlog`, `Retry-After: 60`, and record no state/ledger/revision. ACTIVE resume/same-terminal no-op/exact replay append nothing and remain available. Purge may append at length 1,023, yielding 1,024; if a purge DELETE is already present it reuses it, and no other transition can append while PURGING. A structurally corrupt queue returns 500 rather than capacity. Successful alarm dequeue reopens capacity; no queue entry is evicted or coalesced.
- Every start/progress/terminal/purge transition that appends a projection calls top-level SQLite DO storage writes plus `setAlarm(Date.now())` inside the same explicit `ctx.storage.transaction`; no external I/O occurs inside. Alarm and state/ledger/revision/queue commit atomically. `setAlarm` failure aborts everything, records no mutation id, and returns `500 internal_error`, so client retry is fresh rather than a replay of accepted state. The first alarm is immediate. On D1 head failure, the alarm transaction persists retry count and its next scheduled alarm together; if that transaction or setAlarm fails, the handler throws for Cloudflare's at-least-once retry. A successful retry transaction restores the planned indefinite 2/4/8/16/32/60 schedule. Card GET rearming is additional repair, not the only creation path.
