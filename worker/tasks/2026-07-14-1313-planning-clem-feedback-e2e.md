## Decisions Made
- Shopping restoration is one shared behavior across web, REST v1, and MCP. Re-adding checked or deleted items restores them predictably, while repeated and concurrent mutations preserve one coherent list.
- Authenticated cooking is Cloudflare Durable Object-canonical so progress can continue across devices. D1 remains the owner-private query/history projection, authenticated client storage remains a non-canonical offline queue, and anonymous progress remains local-only.
- Logout/reset proves zero authenticated identity/progress residue and fences authenticated stale writers. Already-loaded legacy tabs may still recreate account-neutral anonymous recipe progress; that state never auto-adopts or syncs and always requires the current user's explicit choice.
- Cook sessions pin a recipe snapshot. Recipe edits never silently remap an active session, completion and abandonment do not create social activity, and owner-authorized purge removes disposable content while retaining only declared arbitration metadata.
- SavedRecipe is private canonical save state, separate from cookbook membership. Search and API projections are viewer-scoped, public counts do not ship, and personalized reads remain private and non-cacheable.
- Native sync leaves the complete schema-1 union and legacy cursor behavior unchanged. Explicit schema 2 adds private savedRecipe changes and a schema-bound opaque cursor; a retained account floor and account-wide D1 monotonic allocator prevent save/unsave/tag changes from hiding behind future-skewed or later-deleted resources.
- Task-owned atomic Cloudflare mutations use one narrow native D1 batch port in Workers. Release tooling may use the D1 REST batch endpoint only after a provider-backed QA qualification proves first/middle/last rollback for that exact release; otherwise release blocks. Prisma remains the ordinary data API but is not used as a production D1 transaction boundary. The existing dual-guard native dogfood harness may use the same port through a real local SQLite transaction.
- Native-sync migration/source parsing accepts only the exact canonical-Z, Prisma-adapter `+00:00`, and SQLite-default UTC timestamp encodings already persisted by the product, normalizes them before comparison, and emits canonical millisecond Z for all new sync values. The Prisma D1 adapter is pinned exactly.
- Additive shopping mutations persist a request-hash receipt and, for REST, the completed idempotent response in the same atomic batch, so post-commit response loss cannot duplicate quantity or restoration effects.
- Shopping mutation identities are explicit in REST, persisted per logical web action, and optional-with-session/request-id fallback in MCP. Receipts expire at 24 hours and are pruned through bounded mutation work plus Cloudflare Cron; no transport may commit a reservation separately from its shopping effect.
- Every native-visible shopping/save/tag mutation allocates from the complete account clock. Native feed repair and all source reads share one D1 batch snapshot, and compatibility catch-up excludes savedRecipe tombstones so an explicit unsave remains authoritative until a new save action.
- Remote D1 REST atomicity is a release prerequisite rather than an assumption: each mutation-bearing release owner consumes a fresh immutable QA qualification artifact bound to exact source/tree, provider/database, endpoint contract, adapter bytes, and AtomicSqlPort contract. Production workflow qualification occurs before production migration, and Unit37 independently requalifies before its remote mutation suffix.
- Manual controlled course plus owner-authored custom tags ship now. AI categorization is deferred until a proposal/review model and deliberate UX exist.
- REST v1 and MCP receive backward-compatible neutral recipe metadata and detail-only scaling. Existing public field types and stored recipe quantities remain unchanged; private isSaved is REST-only and entitlement-gated.
- Current navigation remains unless testing proves an accessibility or reachability defect. Changed product surfaces receive desktop/mobile visual QA; no broad dock redesign ships.
- Import remains agent/API-only. No first-party import surface, Pebble-specific behavior, automatic RecipeSpoon creation, public cook history, or cross-user collaborative cooking ships.
- Delivery is exact-SHA and Cloudflare-first: validate in the Workers runtime, deploy and clean QA before merge, merge only reviewed green code, then verify the automatic production workflow, canonical health, live product/API behavior, and residue cleanup.
- Release execution is evidence-driven and crash-recoverable. Provider outages wait durably rather than becoming code repairs; true code/configuration defects use reviewed repair successors; task/track/archive finality precedes the single truthfully recorded Slugger handoff attempt.
- The exact implementation authority is the doing-owned `./2026-07-14-1313-doing-clem-feedback-e2e/EXECUTION-CONTRACT.md`. It preserves all schemas, CLI vectors, storage bytes, ordering, crash recovery, and terminal protocols formerly embedded here; the doing units own their red/green/verification delivery.
