## Scope

### In Scope
- Make authenticated cook progress canonical in one SQLite-backed Cloudflare Durable Object per user and recipe, with atomic start/resume, same-owner cross-device discovery, and live synchronization.
- Keep anonymous cook progress local-only and prevent authenticated code from reading or uploading anonymous state.
- Pin active cook sessions to the recipe snapshot they started from; support revision-checked progress updates, idempotent completion/abandonment, recipe-edit conflict choices, private resume discovery, projection retry, retention, and owner-authorized purge.
- Store only a complete private registry/discovery projection for unpurged cook objects in D1; never store step, checklist, scale, or recipe-snapshot progress there.
- Route web, REST v1, MCP, and legacy shared shopping additions through one add/restore behavior: checked or deleted items return unchecked at the fresh end position, while active unchecked items preserve position and accumulate quantity.
- Add canonical private `SavedRecipe` state distinct from cookbook membership and social activity, with one-time migration of existing cookbook-derived saved expectations, owner-only paginated search, web controls, and REST v1 endpoints.
- Add recipe-owned manual tags: one controlled course and up to ten custom labels, authoring/editing, bounded filters on My Recipes/search, public read metadata, and REST/MCP read parity.
- Add optional read-time recipe scaling to REST detail and MCP `get_recipe` through one pure shared helper without changing stored quantities or existing field types.
- Add one forward-compatible numeric D1 migration for the new data models, one-time save backfill, and shopping identity repair/index; keep the existing Prisma migration mirror and unrelated migration tooling unchanged.
- Add a dedicated official Workers Vitest project and CI command for the DO/Worker paths, upgrading Vitest to `^4.1.0` and locking a compatible Workers pool, with Istanbul coverage and serialized shared-storage WebSocket execution.
- Add the `CookSession` namespace through the supported legacy `migrations` array using `new_sqlite_classes`. Ship the inert namespace/test/deploy bootstrap in its own reviewed PR with one atomic production deploy, then retain the existing version-upload/canary/promotion path for the product PR and later code-only releases.
- Remove existing Pebble-specific runtime categorization and fixtures while preserving neutral API-provider behavior.
- Preserve current navigation unless implementation reveals a concrete accessibility defect; validate every changed surface on mobile and desktop.
- Ship through the repository's existing CI, QA, pull-request, merge, and automatic production workflow, with exact deployed-revision health checks and disposable-data cleanup.

- AI tag suggestions, inference jobs, confidence/provenance storage, or proposal-review UI.
- Pebble-specific fields, routes, names, telemetry categories, documentation, or behavior.
- Cross-user collaborative cook sessions, public cook history, or automatic `RecipeSpoon` creation.

## Decisions Made
- Authenticated cook state is Durable Object-canonical; anonymous state is browser-local; D1 is metadata-only.
- The SQLite-backed `CookSession` class uses Cloudflare's fully supported legacy lifecycle: explicit top-level and QA bindings plus retained migration `v1_cook_session` with `new_sqlite_classes`. Declarative `exports` is deferred because its permanent prohibition on version upload would remove Spoonjoy's existing canary controls without providing a required capability.
- The bootstrap PR contains exported class `CookSession`, binding `COOK_SESSIONS`, top-level/QA `v1_cook_session` `new_sqlite_classes` migration, `wrangler.workers-test.json`, `vitest.workers.config.ts`, Vitest `4.1.10`, Workers pool `0.18.6`, and a source-controlled one-time atomic production path. It proves the lifecycle boundary before the product PR enables endpoints; failure before boundary verification is repaired forward. The product PR restores the existing canary path, and DO protocols remain forward/backward compatible during gradual deployments.
- Exact same-origin `/api/cook-sessions/:recipeId/*` HTTP/WebSocket endpoints authenticate in the Worker, validate `Origin` for mutations/upgrades, derive the DO with `idFromName(JSON.stringify([userId, recipeId]))` server-side, and return `private, no-store`; WebSocket upgrade responses bypass response cloning.
- Private `GET /api/cook-sessions` reads only the authenticated owner's D1 active rows ordered by `updatedAt DESC, recipeId DESC`. The home polls it every five seconds only while its initial list is empty and the page is visible/online, with immediate revalidation on mount, focus, visibility, and online events.
- The recipe-scoped contract is `POST .../start`, `GET/PATCH .../:recipeId`, `POST .../:recipeId/{complete|abandon|restart}`, `GET .../:recipeId/socket`, and owner-only `DELETE .../:recipeId` for purge. The Worker builds start/restart snapshots from D1; clients never submit trusted recipe content.
- Snapshot version 1 pins title, servings, ordered step IDs/numbers/titles/descriptions/durations, ingredient IDs/names/quantities/units, and step-output-use IDs/links plus a deterministic content hash. The DO stores it with `active|completed|abandoned` state.
- The DO mints `attemptId`; every mutation carries `attemptId`, `expectedRevision`, and `mutationId`. Duplicate mutation IDs replay their result, stale attempts/revisions return 409 with the canonical revision/snapshot, simultaneous starts resume one active attempt, and restart creates a new attempt only by atomically abandoning the old one. Recipe mismatch offers `continue original` or `abandon and start current`; no command silently remaps IDs.
- Canonical state and a highest-revision pending projection persist before D1 writes. Start returns success only after a conditional initial projection; later failures retry monotonically without resurrecting terminal rows. One persisted scheduler owns projection retry and 24-hour terminal retention, then closes sockets, deletes its alarm/storage, and removes the registry row.
- D1 is the complete registry for every unpurged object. Test/smoke cleanup enumerates owner rows, purges each DO first, verifies projection removal, then deletes disposable owner data; partial failures remain retryable and emit bounded privacy-safe errors through existing observability.
- Shopping uses one SQLite `INSERT ... ON CONFLICT ... DO UPDATE ... RETURNING` statement and a partial unique active-identity index over `shoppingListId`, `ingredientRefId`, and `COALESCE(unitId, '')`. The migration merges active duplicates into the lowest `(sortIndex,id)` row, sets its quantity to the sum of non-null quantities or null when all are null, leaves it unchecked when any duplicate is unchecked, and tombstones extras before installing the index. For an active/checked conflict, `(existing,incoming)` quantity is `(null,null)->null`, `(null,n)->n`, `(n,null)->n`, `(n,m)->n+m`; incoming non-null category/icon replaces, incoming null preserves. Active unchecked re-add retains ID/position; checked re-add retains ID, becomes unchecked, and moves to `MAX(active sortIndex)+1`; a deleted row never conflicts, retains its tombstone, and re-add creates a new ID at that same fresh-end rule with only incoming quantity/category/icon. Each success sets `updatedAt` strictly greater than the row's prior timestamp using SQL's max of bound now and prior +1ms. No application retry risks double-add; a failed statement leaves no partial write and surfaces the existing adapter error.
- SavedRecipe key is `(userId, recipeId)` with `savedAt`; backfill deduplicates memberships by `Cookbook.authorId` and sets `savedAt = MAX(RecipeInCookbook.updatedAt)`. Afterward only explicit save/unsave changes it. REST uses `GET /api/v1/saved-recipes`, idempotent `PUT`/`DELETE /api/v1/saved-recipes/:recipeId`, existing kitchen scopes/idempotency, and `Cache-Control: private, no-store` plus credential-varying headers. Lists order by `savedAt DESC, recipeId DESC`; a versioned base64url JSON cursor carries both values and applies the matching strict-less-than predicate, with malformed cursors returning 400.
- The product schema is frozen as follows. Add `User.savedRecipes: SavedRecipe[]`, `User.cookSessionIndexes: CookSessionIndex[]`, `Recipe.savedBy: SavedRecipe[]`, and `Recipe.tags: RecipeTag[]`. `Recipe.course` is nullable `String` with no default, raw `CHECK (course IS NULL OR course IN ('main','side','appetizer','dessert'))`, and modeled index `(course, deletedAt, updatedAt)`. `SavedRecipe` fields are `userId String`, `recipeId String`, and `savedAt DateTime @default(now())`; it has composite primary key `(userId, recipeId)`, `user`/`recipe` relations with cascading deletes, and indexes `(userId, savedAt, recipeId)` plus `(recipeId)`. `RecipeTag` fields are `id String @id @default(cuid())`, `recipeId String`, `label String`, `normalizedLabel String`, `createdAt DateTime @default(now())`, and `updatedAt DateTime @default(now()) @updatedAt`; `recipe` cascades on delete, `(recipeId, normalizedLabel)` is unique, and `(normalizedLabel, recipeId)` is indexed. `CookSessionIndex` fields are `userId String`, `recipeId String`, `attemptId String`, `status String`, `revision Int`, `title String`, `startedAt DateTime`, `updatedAt DateTime`, and `terminalAt DateTime?`, all without defaults; it has composite primary key `(userId, recipeId)`, `user` cascading on delete, deliberately no recipe relation, raw checks `status IN ('active','completed','abandoned')`, `revision >= 0`, and `length(title) <= 200`, and indexes `(userId, status, updatedAt, recipeId)` plus `(status, terminalAt)`. It has no snapshot/progress/checklist/scale columns. Exact index names are `Recipe_course_deletedAt_updatedAt_idx`, `SavedRecipe_userId_savedAt_recipeId_idx`, `SavedRecipe_recipeId_idx`, `RecipeTag_recipeId_normalizedLabel_key`, `RecipeTag_normalizedLabel_recipeId_idx`, `CookSessionIndex_userId_status_updatedAt_recipeId_idx`, `CookSessionIndex_status_terminalAt_idx`, and raw `ShoppingListItem_active_identity_key`; migration 0024 drops only `ShoppingListItem_shoppingListId_unitId_ingredientRefId_key` before creating it.
- Saved REST responses use the existing `{ok, requestId, data}` envelope. `GET /api/v1/saved-recipes?q=&limit=&cursor=` returns `{recipes: Array<RecipeSummary & {savedAt}>, nextCursor}`; searchable fields are recipe title, description, chef username, course, and tag label, compared case-insensitively after trimming/collapsing `q`. The unpadded base64url cursor decodes exactly to `{v:1,savedAt:<ISO string>,recipeId:<string>}`. `PUT .../:recipeId` accepts only `{clientMutationId}`, returns 201 on first save or 200 on an existing save with `{saved:true,recipeId,savedAt,mutation}`; `DELETE` accepts the same body and always returns 200 with `{saved:false,recipeId,mutation}`. Missing/soft-deleted recipes return existing `not_found`; invalid query/cursor/body returns existing `validation_error` with a `field` detail.
- Neutral metadata is top-level `course: string|null` and `tags: string[]` on recipe detail and summary objects produced by `recipeDetail`, `formatRecipe`, and `formatRecipeSummary`, including REST recipe list/detail/search and MCP `list_recipes`/`get_recipe`/`search_spoonjoy`. Web display/filter scope is recipe detail, My Recipes, and global search only. Optional scaling applies only to REST `GET /api/v1/recipes/:id?scale=` and MCP `get_recipe({scale})`; when present it adds top-level `scale:{factor,appliedTo:'ingredient_quantities',decimalPlaces:6}` and scales each ingredient `quantity`, while absent scale is byte-shape compatible. Invalid scale maps to REST `validation_error` field `scale` and MCP's existing invalid-argument error.
- Cook HTTP is frozen to `GET /api/cook-sessions`, `POST /api/cook-sessions/:recipeId/start`, `GET|PATCH|DELETE /api/cook-sessions/:recipeId`, `POST .../:recipeId/{complete|abandon|restart}`, and `GET .../:recipeId/socket`. Success is `{state}` for detail/mutations, `{sessions}` for list, and empty 204 for purge; start returns 201 when created or 200 when resumed, all other state mutations return 200. Errors are `{error:{code,message,retryable,state?}}` with 400 `invalid_request`, 401 `authentication_required`, 403 `origin_forbidden`, 404 `not_found`, 409 `stale_attempt|stale_revision|recipe_changed`, and 503 `projection_unavailable|purge_incomplete`. Mutation/restart/terminal bodies contain exactly `attemptId`, `expectedRevision`, and `mutationId`; PATCH also contains nonempty `changes` from `activeStepIndex`, `scaleFactor`, `checkedIngredientIds`, and `checkedStepOutputIds`. IDs must exist in the pinned snapshot, arrays are unique, step index is in range, and cook scale preserves the current UI bound `0.25..50`.
- Cook state is `{version:1,recipeId,attemptId,status,revision,snapshotHash,snapshot,progress,startedAt,updatedAt,terminalAt}` with ISO timestamps. Snapshot is `{version:1,recipe:{id,title,servings},steps:[{id,stepNum,stepTitle,description,duration,ingredients:[{id,name,quantity,unit:{id,name}}],usingSteps:[{id,inputStepNum,outputStepNum}]}]}`; nullable source values stay null. Steps sort by `(stepNum,id)`, ingredients by `id`, and uses by `(inputStepNum,outputStepNum,id)`. `snapshotHash` is lowercase SHA-256 hex of UTF-8 `JSON.stringify` over that fixed key order. Progress is `{activeStepIndex,scaleFactor,checkedIngredientIds,checkedStepOutputIds}`. List items are exactly `{recipeId,attemptId,status:'active',revision,title,startedAt,updatedAt}`; snapshot title is full length, while the D1 title projection truncates to 200 Unicode code points.
- The SQLite DO schema is exact: `session(singleton INTEGER PRIMARY KEY CHECK(singleton=1),version INTEGER,recipe_id TEXT,attempt_id TEXT,status TEXT,revision INTEGER,snapshot_hash TEXT,snapshot_json TEXT,active_step_index INTEGER,scale_factor REAL,checked_ingredient_ids_json TEXT,checked_step_output_ids_json TEXT,started_at INTEGER,updated_at INTEGER,terminal_at INTEGER NULL,purge_at INTEGER NULL)`; `mutation_receipt(attempt_id TEXT,mutation_id TEXT,request_hash TEXT,base_revision INTEGER,result_revision INTEGER,response_json TEXT,created_at INTEGER,PRIMARY KEY(attempt_id,mutation_id))`; and `pending_projection(singleton INTEGER PRIMARY KEY CHECK(singleton=1),revision INTEGER,payload_json TEXT,attempt_count INTEGER,next_attempt_at INTEGER)`. Integer times are Unix milliseconds. Request hash is lowercase SHA-256 hex of fixed-key-order UTF-8 `JSON.stringify({operation,body})`. Receipts live for the attempt and 24-hour terminal window. Projection retry delays are 1s, 5s, 30s, 120s, then 600s capped; one alarm schedules the earlier of projection retry and `terminalAt + 24h`, with due purge first. Purge closes sockets, deletes D1 row, then alarm/SQLite storage; D1 failure preserves storage and retries after 60s.
- WebSockets are server-to-client snapshot fan-out only. On connect and every accepted HTTP mutation they send `{type:'snapshot',state}`; protocol errors use `{type:'error',error}` before closure. Any client data message receives error code `client_messages_unsupported`, message `Cook session sockets are receive-only.`, retryable false, then closes code 1003/reason `client-messages-unsupported`. Restart closes old-attempt sockets with 4009/`stale-attempt`; terminal/purge close with 1000 and `session-terminal`/`session-purged`. Browser reconnect delays are 1s, 2s, 5s, then 10s capped while mounted/online. Projection/purge telemetry is limited to event name plus `operation`, `status`, clamped `attemptCount` `0..1000`, `retryDelayMs` `0..600000`, and `revision`; it never includes user/recipe/attempt IDs, title, snapshot, progress, or request bodies.

Binding is `COOK_SESSIONS`; the Worker derives `bootstrap:<CF_VERSION_METADATA.id>` and sends `POST https://cook-session.internal/__bootstrap/probe`, header `X-Spoonjoy-Internal-Probe: 1`, body `{\"version\":1}`. The DO accepts only that exact request, creates/inserts/selects/drops `__bootstrap_probe`, calls `deleteAll`, verifies zero user tables, and returns `{ok:true,storage:'sqlite',residue:0}`; Worker adds `workerVersionId` and the normal version header.
