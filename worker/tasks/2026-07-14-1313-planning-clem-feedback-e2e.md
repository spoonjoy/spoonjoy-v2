## Decisions Made

**Shopping-List State Matrix**

| Existing row | Quantity after add | Checked state | Ordering | Category/icon |
| --- | --- | --- | --- | --- |
| No row | Requested quantity exactly, including `null` | Unchecked; `checkedAt` and `deletedAt` null | Newly allocated end position | Per field, explicit string or null stores exactly; omitted infers from ingredient name through `resolveIngredientAffordance` |
| Active, unchecked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Unchecked | Unchanged | Preserve unless request explicitly overrides |
| Active, checked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Reset unchecked; clear `checkedAt` | Move to a newly allocated end position | Preserve unless request explicitly overrides |
| Deleted tombstone | Requested quantity exactly, including `null`; never add stale quantity | Reset unchecked; clear `checkedAt` and `deletedAt` | Move to a newly allocated end position | Per field, explicit string or null stores exactly; omitted preserves stored non-null and infers only stored-null through `resolveIngredientAffordance` |

Recipe batch adds aggregate equal ingredient-reference/unit pairs before mutation: numeric requested quantities sum, while an all-null group remains null. Web, REST, and MCP call the same D1-safe helper.

Every post-migration active item obeys `checked === (checkedAt !== null)`. Other item commands are exact: setting an unchecked active item checked captures the batch timestamp, sets both fields, and moves it to one newly allocated end slot; setting a checked active item unchecked clears both fields and preserves sortIndex; requesting the already-canonical state is a no-op. Single delete preserves item content/order/check fields and sets deletedAt on an active item; an already-deleted target is a no-op. Clear-completed applies that delete transition to every active checked item; clear-all applies it to every active item. Deletes never compact or rewrite surviving sortIndex values. An empty aggregate, missing eligible target, or clear with zero eligible rows changes no row, version, allocator, or timestamp. The old `normalizeShoppingListOrdering` write path is removed rather than treated as a reorder command; this release adds no user reorder API.

All content-changing shopping-list commands use an atomic list-level lease/version batch. `ShoppingList` adds `mutationVersion Int @default(0)`, nullable `mutationLease String?`, and `nextSortIndex Int @default(0)`. Empty aggregate add, set-checked/delete against a missing list/target, and clear against a missing/empty list return no-op before any list INSERT. Only a fully validated add/free-text/recipe aggregate with at least one row that can create or restore content may bootstrap. It uses one parameterized `INSERT ... ON CONFLICT(authorId) DO NOTHING` with a caller-generated nonempty id, version/index zero, null lease, and database timestamps; concurrent losers read the winner, and bootstrap itself never creates an item or consumes version/index. Every attempt reads that ShoppingList row/version `V` first, then its item snapshot sequentially. It computes the exact changed set/reserved count and returns immediately for a no-op. Otherwise one non-interactive D1 transaction conditionally acquires a random lowercase UUID-v4 lease only where `mutationVersion = V AND mutationLease IS NULL`, guards every item statement and contiguous allocation on that lease, then clears the lease and increments version. A concurrent commit visible during or after item read makes the version CAS fail, every guarded statement no-op, and the request retry from a fresh version-first read. Eight attempts use fixed inter-attempt delays 5, 10, 20, 40, 80, 160, and 320 ms; exhaustion throws `ShoppingListBusyError`. REST maps it to additive `409 write_conflict` / `Shopping list changed too frequently; retry.`, web returns that message with 409, and MCP returns a retryable tool error with that text. Every add/set-checked/delete/clear web, v1, MCP, and recipe writer uses this coordinator.

Equal ingredient-reference/unit pairs aggregate in original request order. Numeric quantities sum and an all-null group stays null. Category/icon use independent three-state input. Explicit string or null always stores exactly; the last explicit aggregate value per field wins. Omitted on an active row preserves that field even when null. Omitted on a missing row infers that field from canonical ingredient name through `resolveIngredientAffordance`. Omitted on a tombstone preserves a stored non-null field but infers only a stored-null field; it never recomputes a non-null tombstone affordance. The accepted item batch is all-or-nothing. `nextSortIndex` reserves one contiguous range in aggregate input order; abort/retry never consumes it. Successful check/add/restore moves and deletes may leave gaps. Concurrent numeric adds sum, null adds retain serialization-current quantity, simultaneous tombstone restores become first restore plus subsequent active add, and later commit wins conflicting overrides; tests force both D1 commit orders.

One accepted content-changing batch captures one safe `acceptedAtMs = max(platformNowMs, ShoppingList.updatedAtMs + 1, each changed existing item's updatedAtMs + 1)` before SQL and range-checks it against the JavaScript Date limit. Every created/changed item and the parent ShoppingList receive exactly its canonical ISO timestamp; untouched rows do not. A no-op, failed capacity check, lost lease, aborted transaction, or retry changes no timestamp. This monotonic list clock makes REST/native sync cursors observe every accepted mutation even when multiple changed items share a timestamp; item-id tie order remains authoritative inside one batch.

Nullable-unit identity is database-enforced without a sentinel collision. Migration 0024 groups by actual `(shoppingListId, ingredientRefId, unitId)` so SQL NULLs group together; the existing Prisma composite unique continues to protect non-null unit IDs, and exact partial unique index `ShoppingListItem_list_ingredient_null_unit_key` protects `(shoppingListId, ingredientRefId) WHERE unitId IS NULL`. Before reconciliation it normalizes every legacy row by canonical checked value `checked OR checkedAt IS NOT NULL`: false/null stays unchecked; false/non-null becomes checked preserving checkedAt; true/non-null remains; true/null becomes checked with checkedAt=updatedAt. Within each duplicate group order is active before tombstoned, then sortIndex, updatedAt, id. If active rows exist, the first is survivor; its quantity is the finite sum of active non-null quantities or null when all are null, it is active, it is unchecked/null if any active row is canonical-unchecked, otherwise checked with the first active checkedAt in order, and category/icon are the first non-null active values in order. If all are tombstones, the first normalized row survives unchanged. Every nonsurvivor is deleted.

Migration 0024 creates durable `ShoppingListItemSyncTombstone` before reconciliation so deleting a duplicate never creates a client-side ghost. Its exact no-extra-purpose row is the original loser id plus `shoppingListId`, preserved normalized `quantity`, `unitName`, `ingredientName`, `checked`, `checkedAt`, `sortIndex`, `categoryKey`, and `iconKey`, with `deletedAt` equal to the loser's existing deletion time or the account's migrationSyncAt for an active loser, and `updatedAt` equal to migrationSyncAt. It has a cascading ShoppingList relation, primary key `id`, and cursor index `(shoppingListId,updatedAt,id)`. Before any write, migration 0024 captures one database-now millisecond and computes per affected account/list `syncHighWaterMs` as the exact maximum of that value and every timestamp that either pre-migration sync endpoint can issue in a cursor: `User.updatedAt`, `ShoppingList.updatedAt`, every ShoppingListItem.updatedAt, every active Recipe.updatedAt, every Cookbook.updatedAt, and every NativeSyncTombstone.updatedAt for the author. Every source timestamp must parse to a canonical finite JavaScript-Date millisecond. Exact `migrationSyncAtMs=syncHighWaterMs+1` must remain within the JavaScript Date limit; parse/range/overflow failure aborts the whole migration before schema/data writes. This dominates equal and future-skewed stored timestamps from both shopping and native account streams, including a native cursor last advanced by another resource kind. The migration copies every nonsurvivor into the ledger before hard deletion, and rerun neither duplicates nor retimestamps a row. `GET /api/v1/shopping-list/sync` merges live/soft-deleted items with these ledger rows, maps a ledger row to the existing deleted shopping-item payload, and orders/pages the combined stream by `(updatedAt,id)`. `GET /api/v1/me/sync` merges the same ledger rows as `action:'delete',kind:'shoppingItem'` with exact parent/title/deletion metadata and its existing `(updatedAt,kind,id)` cursor order. Ordinary shopping-list reads exclude the ledger. Route tests prove legacy ISO and opaque composite cursors at the former high-water timestamp, tied ids on either side of the cursor, and a native cursor advanced by a future-skewed profile/recipe/cookbook/tombstone all receive every deleted duplicate id through both sync APIs; a cursor at migrationSyncAt does not replay them, pagination neither drops nor duplicates an id, rerun preserves cursor bytes/timestamps, and user/list deletion cascades the ledger.

Before any migration write, SQL guards prove the sync high-water/date rules above, every existing sortIndex is an integer in `0..2_147_483_646`, and every non-null duplicate active-quantity sum is finite and in `[-Number.MAX_VALUE, Number.MAX_VALUE]`; failure rolls back the whole D1 migration and leaves original rows/schema. `nextSortIndex` initializes to zero for empty lists or exact widened `MAX(sortIndex)+1` otherwise, so `2_147_483_647` is the exhausted sentinel and never an item sortIndex. Reconciled survivors, every new loser tombstone, and parent lists get that account's exact migrationSyncAt so both cursor streams see all changes. Migration rerun is harmless.

Three-state inputs are exact before aggregation. REST JSON and MCP: absent/`undefined` means omitted, JSON/tool `null` means explicit null, and any string including `""` means that exact explicit string; other types and unknown fields are validation errors. Web FormData: zero entries means omitted, exactly one string entry including `""` is explicit, a File or repeated field is `400`; literal `null` is a string. Recipe ingredient batches and free-text creation pass omitted affordances after canonical ingredient-name resolution. Quantity inputs/results must be finite and in `[-Number.MAX_VALUE, Number.MAX_VALUE]`; overflow maps to exact `Quantity is outside supported numeric range.` Global IngredientRef/Unit get-or-create happens idempotently before the list transaction and is explicitly outside item-batch atomicity: a later busy/capacity failure may leave an unreferenced canonical dictionary row but never partial user/list state.

Shopping `mutationVersion`/`nextSortIndex` are `0..2_147_483_647`; item sortIndex is `0..2_147_483_646`. A content change requires `mutationVersion < 2_147_483_647`, every retained sortIndex in range, and widened `nextSortIndex + reservedCount <= 2_147_483_647`; allocated indexes are `[nextSortIndex,nextSortIndex+reservedCount-1]`, then the stored next value advances by reservedCount. Thus allocating index `2_147_483_646` legally stores the exhausted sentinel, and any later allocation fails without writes. Capacity failure throws `ShoppingListCapacityError`; REST returns `409 capacity_exceeded` / `Shopping list numeric capacity reached.`, web returns 409 with that message, and MCP returns a non-retryable tool error. Lease retries never mask capacity/quantity errors.

Exact `stale_attempt`, `stale_revision`, and `session_purging` errors occur only while a current snapshot exists and require `data:{snapshot}` equal to that current state; every other error forbids `data`. A bodyless 204 still carries `X-Request-Id`, `Cache-Control: private, no-store`, and `Pragma: no-cache` but no envelope/body. An inbound trimmed `X-Request-Id` matching `^[A-Za-z0-9._:-]{1,128}$` is echoed; otherwise generate `req_<lowercase UUID-v4>`. The response header and envelope value are identical. Every other non-101 response has the same private headers.
- `POST /api/cook-sessions` accepts exact JSON `{ recipeId, mutationId, adoption? }`; unknown fields and explicit `adoption: null` are 422. `adoption`, when present, is exactly `{ version: 1, recipeFingerprint, activeStepIndex, scaleFactor, checkedIngredientIds, checkedStepOutputIds }`; local anonymous `updatedAt` is never transmitted, accepted, or hashed. Absent adoption is `not_requested`. Otherwise return `503 projection_backlog`, `Retry-After: 60`, and record no state/ledger/revision. ACTIVE resume/same-terminal no-op/exact replay append nothing and remain available. A previously accepted queue therefore guarantees purge can append at length at most 1,023, yielding at most 1,024; an existing purge DELETE is reused, and no other transition can append while PURGING. A structurally valid but already oversized queue is corruption/500 rather than capacity. Successful alarm dequeue reopens count/byte capacity; no entry is evicted or coalesced.
- Every start/progress/terminal/purge transition that appends a projection calls top-level SQLite DO storage writes plus `setAlarm(Date.now())` inside the same explicit `ctx.storage.transaction`; no external I/O occurs inside. Alarm and state/ledger/revision/queue commit atomically. `setAlarm` failure aborts everything, records no mutation id, and returns `500 internal_error`, so client retry is fresh. The first alarm is immediate. One FIFO-gated alarm invocation validates all persisted state, then independently runs every due channel: it processes at most 32 projection heads sequentially and, while current is PURGING, performs one complete socket sweep when the socket channel is due. For each projection head it performs exactly one idempotent D1 transaction, then one DO transaction verifies/removes that exact unchanged head and clears only `retry-state.projection`; it may continue through head 32. A socket sweep attempts PURGING send then close for every attached socket independently; zero remaining sockets clears only `retry-state.sockets`, while any thrown operation or remaining socket advances only that channel. Projection and socket work due in the same alarm are both attempted even when the other fails. After both channels, PURGING finalization runs only when its DELETE barrier is gone and the socket set is empty. The next alarm is immediate when a projection queue remains after 32, otherwise the minimum of both retry channels and coordinator/tombstone metadata deadlines; no remaining work deletes the alarm. D1 success followed by DO-commit failure leaves the head and relies on D1 replay no-op under Cloudflare alarm retry.
- Projection and socket channels use independent consecutive-failure transitions. First through fifth failures for that channel persist `failures=1..5` and delays 2/4/8/16/32 seconds; when its stored failures is already five, every later failure keeps five and delays 60 seconds indefinitely. A successful projection head clears only projection retry before a later head can establish a fresh sequence; a successful socket sweep that reaches zero clears only socket retry. Initial purge fanout that leaves/throws on any socket establishes socket failure one even if projection work also fails. Each head-removal transaction is already durable before the next head; the final retry-state/alarm transaction is computed from the post-work queue/socket state and atomically updates both channels plus the one minimum alarm. If that final transaction fails, prior successful head removals and their per-head projection-channel clears remain valid; the thrown alarm retries from the newly persisted queue and last committed retry state without inventing either channel.
