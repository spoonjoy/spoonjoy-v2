## Product And Data Contracts

### Shopping-List State Matrix

| Existing row | Quantity after add | Checked state | Ordering | Category/icon |
| --- | --- | --- | --- | --- |
| No row | Requested quantity exactly, including `null` | Unchecked; `checkedAt` and `deletedAt` null | Newly allocated end position | Per field, explicit string or null stores exactly; omitted infers from ingredient name through `resolveIngredientAffordance` |
| Active, unchecked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Unchecked | Unchanged | Preserve unless request explicitly overrides |
| Active, checked | If requested is `null`, retain existing; otherwise `(existing ?? 0) + requested` | Reset unchecked; clear `checkedAt` | Move to a newly allocated end position | Preserve unless request explicitly overrides |
| Deleted tombstone | Requested quantity exactly, including `null`; never add stale quantity | Reset unchecked; clear `checkedAt` and `deletedAt` | Move to a newly allocated end position | Per field, explicit string or null stores exactly; omitted preserves stored non-null and infers only stored-null through `resolveIngredientAffordance` |

Recipe batch adds aggregate equal ingredient-reference/unit pairs before mutation: numeric requested quantities sum, while an all-null group remains null. Web, REST, and MCP call the same D1-safe helper.

All shopping-list item writes use an atomic list-level lease/version batch. `ShoppingList` adds `mutationVersion Int @default(0)`, nullable `mutationLease String?`, and `nextSortIndex Int @default(0)`. Every attempt reads the ShoppingList row and observes version `V` first, then reads its items; those reads are sequential and never parallel or items-first. It then submits one non-interactive D1 transaction of parameterized guarded raw-SQL statements: conditionally acquire a random-UUID lease only where `mutationVersion = V AND mutationLease IS NULL`, run every item/create/delete/reorder statement only where that lease exists, allocate the required sort range, then clear the lease and increment version. Because the version read precedes the item read, any concurrent commit visible during or after the item read makes the later version CAS fail; a request can commit only from item state associated with its observed version. If acquisition changes zero rows, every guarded statement is a no-op and the whole request retries from a fresh version-first read. Eight attempts are allowed with fixed delays 5, 10, 20, 40, 80, 160, and 320 ms; exhaustion throws `ShoppingListBusyError`. REST maps it to additive `409 write_conflict` / `Shopping list changed too frequently; retry.`, the web action returns the same 409 message, and MCP returns a retryable tool error with that text. Migration `0024_shopping_list_mutation_coordination.sql` initializes `nextSortIndex` to each list's current `MAX(sortIndex)+1` (zero for empty), version zero, and null lease. Every item mutation uses this coordinator.

Equal ingredient-reference/unit pairs aggregate in original request order. Numeric quantities sum and an all-null group stays null. Category/icon use independent three-state input. Explicit string or null always stores exactly; the last explicit aggregate value per field wins. Omitted on an active row preserves that field even when null. Omitted on a missing row infers that field from canonical ingredient name through `resolveIngredientAffordance`. Omitted on a tombstone preserves a stored non-null field but infers only a stored-null field; it never recomputes a non-null tombstone affordance. Web, REST, MCP, and recipe batches pass the same sentinel semantics. The accepted batch is all-or-nothing. `nextSortIndex` reserves one contiguous range in aggregate input order; abandoned retry ranges cannot commit, while later successful requests may leave gaps only after an already-committed allocation and later unrelated delete. Concurrent numeric adds sum, null adds retain serialization-current quantity, simultaneous tombstone restores become first restore plus subsequent active add, and later commit wins conflicting overrides. D1 commit order is the only winner order; tests assert both request orders.

### Cook-Session Ownership And Lifecycle
- Every JSON cook response carries `X-Request-Id` and uses one exact envelope. Success is `{ ok: true, requestId, data }`. Error is `{ ok: false, requestId, error: { code, message, status }, data? }`; `data` is present as `{ snapshot }` on stale attempt/revision conflicts when a current attempt exists. An inbound trimmed `X-Request-Id` matching `^[A-Za-z0-9._:-]{1,128}$` is echoed; otherwise generate `req_<UUID-v4>`. The response header and envelope value are identical. Every non-101 response sets `Cache-Control: private, no-store` and `Pragma: no-cache`.
- `POST /api/cook-sessions` accepts exact JSON `{ recipeId, mutationId, adoption? }`; unknown fields and explicit `adoption: null` are 422. `adoption`, when present, is exactly `{ version: 1, recipeFingerprint, activeStepIndex, scaleFactor, checkedIngredientIds, checkedStepOutputIds }`; local anonymous `updatedAt` is never transmitted, accepted, or hashed. The persisted start command stores `adoption: AdoptionInput|null`; null encodes wire omission. A newly created attempt returns 201; active resume/replay returns 200; data is `{ snapshot, adoptionOutcome }` with exact `not_requested|adopted|server_won`. Absent adoption is `not_requested`. Every supplied object first requires exact fields/version, 64-hex recipeFingerprint, integer index, finite scale 0.25..50, and nonempty-string duplicate-free arrays; structural failure returns 422 before state lookup. Pristine first start also requires current fingerprint/index/subsets; applied state arrays are canonicalized in pinned traversal order, success is `adopted`, and semantic failure is 422. With any attempt/history, structurally valid adoption is ignored without recipe/pinned identity comparison and returns `server_won`. Idempotency hashing is state-independent: for every present adoption, both checked arrays sort lexicographically before recursively sorted-key canonical JSON hashing of exact `{ recipeId, adoption }`; omitted adoption remains absent, and mutation/request ids plus client time are excluded. Exact replay therefore matches before/after state creation and returns its recorded outcome plus current snapshot. The separate 64-entry/24-hour replay/conflict/tombstone/backpressure rules remain.
- The Worker derives the canonical DO address as `idFromName("${userId}:${recipeId}")`. That deterministic per-user/per-recipe object, not D1, arbitrates start/resume and guarantees one active attempt. Parallel starts serialize in the same object.
- Every new attempt gets a random `attemptId` UUID stored inside the DO and returned in snapshots. D1 stores one history/index row per attempt and retains nullable unique `activeKey = "${userId}:${recipeId}"` as a projection integrity check, but start/resume never consults D1 to decide canonical state.
- Active recipes are cookable by any authenticated user because the current product has public active recipes. Creating any new attempt, including terminal-to-new restart, uses the same `deletedAt: null` visibility rule as recipe detail; missing, soft-deleted, or future visibility-inaccessible recipes all return indistinguishable `404 not_found`, and the terminal/current coordinator state is retained unchanged. No recipe-ownership restriction is introduced. Once created, the pinned attempt remains GET/PATCH/live/complete/abandon/purge accessible to its owner without re-reading Recipe visibility, and active start/resume returns it, even if the recipe is later edited, soft-deleted, or visibility-inaccessible while its row remains; uncoordinated hard deletion is outside this guarantee. A soft-deleted recipe cannot create another attempt. The recipe route falls back to the authenticated caller's own current pinned snapshot when public recipe lookup fails, rendering cook-only content; anonymous callers and authenticated callers without their own attempt receive the ordinary 404.
- `GET /api/cook-sessions/recipes/:recipeId` returns `200 { snapshot }` from the authenticated user's deterministic DO. `PATCH` accepts `{ attemptId, baseRevision, mutationId, operations }`, with one to 64 operations from exactly: `{ type: "set_active_step", stepIndex }`, `{ type: "set_scale_factor", scaleFactor }`, `{ type: "set_ingredient_checked", ingredientId, checked }`, and `{ type: "set_step_output_checked", stepOutputId, checked }`. Accepted and replayed mutations return `200 { snapshot }`.
- Idempotency equality uses canonical validated payloads, never raw bytes: reject unknown fields, lexicographically sort adoption checked arrays for hashing, recursively sort object keys, preserve PATCH operation-array order, then hash canonical JSON; request-id/header differences are excluded. Pinned traversal order is used only when applying a pristine adoption to snapshot state, never in the start request hash. PATCH validates every operation before applying any. Operations execute sequentially in array order; duplicate targets are allowed and the last operation wins. One invalid operation rejects the whole request with no state, ledger, projection, revision, or timestamp change.
- Revision-zero defaults are `activeStepIndex: 0`, `scaleFactor: 1`, empty checked arrays, ACTIVE status, null terminal timestamps, and one clock reading used for equal `startedAt`/`updatedAt`. Each accepted non-replayed PATCH increments revision exactly once regardless of operation count or whether final values equal prior values; `updatedAt` becomes `max(clockNow, previousUpdatedAt + 1ms)`. `POST .../complete` and `POST .../abandon` accept `{ attemptId, baseRevision, mutationId }`, increment revision once on the first accepted transition, update time by the same monotonic rule, return `200 { snapshot }`, and enqueue an ordered D1 projection clearing `activeKey`. After replay/id-hash and attempt-id checks, terminal-state precedence comes before base-revision comparison: requesting the opposite terminal target returns `409 terminal_conflict` even when stale; repeating the same target with a fresh id ignores stale base and, when ledger capacity remains, records the no-op response then returns the current snapshot without revision/time change; at capacity it returns `429 mutation_ledger_full`. An ACTIVE attempt with stale base returns `409 stale_revision` before capacity. Completion is private history only and never creates a `RecipeSpoon` or notification.
- The progress/terminal mutation ledger is attempt-scoped, retains up to 256 unexpired UUID-v4 records for 24 hours, and never evicts an unexpired replay guarantee. Replay lookup and request-hash comparison happen first. Only accepted mutations enter the ledger: validation, stale attempt/revision, terminal, capacity, and other rejected requests leave no record. While the attempt remains current, an exact accepted replay returns its recorded status/snapshot without incrementing revision; reuse with another payload returns `409 mutation_conflict`. Replacement atomically converts every unexpired progress/terminal record for the old attempt into a non-content `attempt_replaced` tombstone; purge converts the current attempt's records into `attempt_purged` tombstones. These request-hash/id/attempt/outcome/expiry tombstones remain in coordinator storage for the remainder of 24 hours and return the matching 410 rather than a snapshot. Their natural maximum is 64 start-ledger attempts times 256 records; they are never counted as attempt-bearing content. A new mutation at current-ledger capacity returns `429 mutation_ledger_full`. A received stale revision includes latest snapshot; the client rebases with a fresh UUID/latest revision. After network uncertainty, it first retries the canonical-equivalent original request/id. Unrelated fields merge and reconnecting same-field operation wins when accepted.
- A fresh start mutation while the current attempt is `COMPLETED` or `ABANDONED` atomically creates a new ACTIVE attempt at revision zero and returns `201` with that new snapshot and `server_won` for supplied adoption or `not_requested` when absent. The transaction persists new current/coordinator/start-ledger state, converts every unexpired start record targeting the terminal attempt to a non-content `attempt_replaced` tombstone, and appends the new create projection without discarding older projections; only afterward are sockets attached to the terminal attempt closed with `4000 Attempt replaced`. Old start replays return `410 attempt_replaced` and never resolve to the newer attempt. A start while current is ACTIVE resumes it with `200` and returns `server_won` for supplied adoption or `not_requested` when absent.
- `GET /api/cook-sessions/recipes/:recipeId/live` requires a same-origin WebSocket upgrade. Server frames are exactly `{ type: "snapshot", snapshot }` and `{ type: "error", error: { code, message }, snapshot? }`. The server sends an initial snapshot and broadcasts after accepted progress or terminal transitions. Error frames occur only after accepted upgrade: replacement/wake sends `stale_attempt` plus latest then closes `4000 Attempt replaced`; a purge race sends `session_purging` plus PURGING then closes `4001 Session purged`; missing sends `not_found` then `4000 Attempt unavailable`; state-load failure sends `internal_error` then `1011 Internal error`. Send is awaited before close. Ordinary purge broadcasts PURGING then closes every socket with exact `4001 Session purged` without second error; every alarm/event/wake close retry uses that same code/reason even though coordinator state is still PURGING. Client data closes `1003 Read-only subscription`; normal client close is 1000. Attachments are `{ version: 1, userId, recipeId, attemptId }`.
- Upgrade attachment failure returns ordinary `500 internal_error` before 101. After 101, initial-snapshot send failure attempts close `1011 Initial snapshot failed`; close failure is swallowed and the socket is left for later event validation. For every error frame, attempt the specified close code/reason even if frame send fails; send/close errors never change DO state or an already-selected HTTP/upgrade outcome. Replacement attempts stale frame then 4000 for every old socket independently; failures do not roll back the new attempt and a survivor is revalidated on its next event/wake. Accepted non-purge progress/terminal state and projection enqueue commit before fanout; healthy sockets continue, failed send attempts `1011 Broadcast failed`, failed close is swallowed, HTTP remains 200, and no broadcast retry alarm is scheduled. Purge persists fence/barrier/alarm first, then for every socket attempts PURGING snapshot and 4001 independently regardless of send success; failures never roll back or change DELETE 202. While PURGING, every alarm/event/wake retries 4001 on every attached socket, using the projection retry schedule after close failure or a still-present socket. Purge cannot commit its coordinator tombstone or return 204 until `ctx.getWebSockets()` is empty; silent sockets therefore keep DELETE at 202 and remain visible in residue rather than being forgotten. Clients recover by reconnect initial snapshot/GET.
- `DELETE /api/cook-sessions/recipes/:recipeId` accepts `{ attemptId }` and is not mutation-ledger-backed. Precedence is current state first: while any current attempt exists, only its matching id may purge (or continue PURGING); every other id returns `409 stale_attempt` even if it equals `lastPurgedAttemptId`. A valid purge atomically persists `PURGING`, appends a purge barrier after every older projection, broadcasts that terminal state, and attempts to close existing sockets. While PURGING, start/resume, snapshot, patch, complete, abandon, and new upgrades return `409 session_purging` and append nothing. The alarm drains all earlier projections FIFO, idempotently deletes the current D1 attempt row at the barrier, and repeatedly closes every attached socket. Only after `ctx.getWebSockets()` is empty does it atomically remove pinned recipe, progress, current attempt, response-bearing mutation records, projection queue/retry state, and socket bookkeeping while converting unexpired start/progress/terminal mutation records to non-content `attempt_purged` tombstones. Retained coordinator metadata is only `everStarted`, `lastPurgedAttemptId`, and those 24-hour replay tombstones. They can never recreate/adopt state. A fresh mutation id may start a new empty attempt after purge, but `everStarted` makes supplied adoption resolve `server_won` and absent adoption `not_requested`.
- Failure behavior is fixed: local persistence failure before the PURGING fence leaves state unchanged and returns `500`; broadcast/socket-close failure after persistence never rolls back the fence but retains PURGING and an alarm until no attached socket remains; older-projection D1 failure retains queue/barrier/content; current-row D1 delete failure retains barrier/content; a crash after D1 delete retries that delete idempotently; failed atomic attempt-key deletion/tombstone write retains content/barrier and retries; success is reported only after the tombstone is committed, attempt-bearing keys are absent, and `ctx.getWebSockets()` is empty.

Thus opposite terminal target beats ledger-full, same terminal target at capacity returns ledger-full, stale revision beats ledger-full, and PATCH against terminal returns session_terminal before capacity. ### Cook Snapshot, Offline, And Projection
- `CookRecipeSnapshot` is exactly `{ version: 1, id: string, title: string, coverImageUrl: string|null, steps: Step[] }`. Each step is `{ id: string, stepNum: integer, stepTitle: string|null, description: string, duration: integer|null, ingredients: Ingredient[], stepOutputUses: StepOutputUse[] }`. Each ingredient is `{ id: string, ingredientRefId: string, name: string, quantity: finite number, unit: { id: string, name: string } }`. Each step-output use is `{ id: string, outputStepId: string, outputStepNum: integer, outputStepTitle: string|null }`. Stable ids must be nonempty; all stored title/name/description strings, including empty strings, are preserved. Zero/negative step numbers, output step numbers, and durations are legal pinned data and require no migration or renumbering; activeStepIndex always addresses sorted-array position, never stepNum.
- Snapshot construction sorts steps by `(stepNum, id)`, ingredients within a step by `id`, and step-output uses by `(outputStepNum, id)`. Canonicalization recursively sorts object keys lexicographically; preserves array order, stored Unicode code points, case, and whitespace; includes every schema field and emits nullable values as JSON `null`; never emits `undefined`; rejects non-finite numbers; normalizes `-0` to `0`; and otherwise uses ECMAScript JSON number serialization. The fingerprint is lowercase hexadecimal SHA-256 of the UTF-8 canonical recipe-snapshot JSON. Database timestamps and other non-schema fields are excluded.
- `CookSessionSnapshot` is exactly `{ version: 1, attemptId: UUID-v4 string, recipeId: string, recipeFingerprint: 64-char lowercase hex, recipe: CookRecipeSnapshot, revision: nonnegative integer, status, activeStepIndex: integer, scaleFactor: finite number, checkedIngredientIds: string[], checkedStepOutputIds: string[], startedAt: ISO string, updatedAt: ISO string, completedAt: ISO string|null, abandonedAt: ISO string|null }`. Status is `ACTIVE|COMPLETED|ABANDONED|PURGING`; active-step index is zero for an empty recipe and otherwise `0..steps.length-1`; scale is `0.25..50`; checked arrays are duplicate-free subsets of pinned IDs. ACTIVE has null terminal timestamps; COMPLETED only completedAt; ABANDONED only abandonedAt; PURGING preserves prior terminal shape. Pristine adoption additionally obeys these subset/index rules and pinned traversal order. Server-won adoption obeys only the structural/lexicographic rules in the POST contract because it is never applied.

### CookSessionIndex Projection Schema
- D1 model `CookSessionIndex` has: `attemptId String @id`, `userId String`, `recipeId String`, nullable unique `activeKey String?`, `status String`, `recipeFingerprint String`, `recipeTitle String`, `recipeCoverImageUrl String?`, `activeStepIndex Int`, `scaleFactor Float`, `checkedIngredientCount Int`, `checkedStepOutputCount Int`, `revision Int`, `projectionSeq Int`, `lastProjectionId String @unique`, `startedAt DateTime`, `updatedAt DateTime`, `completedAt DateTime?`, and `abandonedAt DateTime?`; user/recipe foreign keys cascade on hard delete. Indexes are `(userId,status,updatedAt)`, `(userId,recipeId,startedAt)`, and `(recipeId)` plus the unique active key/projection id constraints. It stores no full recipe/progress/checklist content.
- Persisted queue entries are a discriminated union. UPSERT is `{ version: 1, projectionId: "<attemptId>:<sequence>:UPSERT", attemptId, sequence, kind: "UPSERT", payload }`, where payload is exactly `{ userId, recipeId, status: "ACTIVE"|"COMPLETED"|"ABANDONED", recipeFingerprint, recipeTitle, recipeCoverImageUrl: string|null, activeStepIndex, scaleFactor, checkedIngredientCount, checkedStepOutputCount, revision, startedAt, updatedAt, completedAt: string|null, abandonedAt: string|null }`. DELETE is `{ version: 1, projectionId: "<attemptId>:<sequence>:DELETE", attemptId, sequence, kind: "DELETE", payload: { userId, recipeId } }`; PURGING is never UPSERTed. Sequence/revision/count/index fields are nonnegative integers (scale finite in range), fingerprints/ids/status/timestamp nullability match the snapshot, and every timestamp is canonical `Date.toISOString()` text. Initial create sequence/revision is 0; progress/terminal UPSERT uses resulting revision; purge DELETE uses current revision + 1. Enqueue and dequeue validate version, exact fields, projectionId components, immutable owner/recipe/attempt, sequence/payload revision equality for UPSERT, and timestamp/status shape. Any malformed persisted head remains queued, makes HTTP mutations return 500 without state change, and makes `alarm()` throw; it is never dropped or sent to D1. A valid UPSERT applies only when `sequence >= projectionSeq`, sets activeKey only for ACTIVE, and records lastProjectionId; exact/older replay is a no-op. DELETE removes only that attempt row when stored sequence is not newer; missing replay is a no-op. FIFO orders old terminal, new active, and purge across attempts.
- Every start/progress/terminal/purge transition that appends a projection calls top-level SQLite DO storage writes plus `setAlarm(Date.now())` inside the same explicit `ctx.storage.transaction`; no external I/O occurs inside. Alarm and state/ledger/revision/queue commit atomically. `setAlarm` failure aborts everything, records no mutation id, and returns `500 internal_error`, so client retry is fresh rather than a replay of accepted state. The first alarm is immediate. On D1 head failure, the alarm transaction persists retry count and its next scheduled alarm together; if that transaction or setAlarm fails, the handler throws for Cloudflare's at-least-once retry. A successful retry transaction restores the planned indefinite 2/4/8/16/32/60 schedule. Card GET rearming is additional repair, not the only creation path.

### RecipeTag

Supporting indexes are `(recipeId, kind, normalizedLabel, id)`, `(kind, normalizedLabel, recipeId)`, and `(createdById)`. Hard-deleting either parent removes tags; soft-deleting a recipe preserves but hides them.
- `kind` is `COURSE` or `CUSTOM`; `source` is `MANUAL` in this release. Only the recipe owner may mutate tags. Manual tags are canonical immediately and have no acceptance transition.
- Course is optional and singular; allowed normalized/storage values are `main`, `side`, `appetizer`, and `dessert`. Custom display labels trim ends, collapse every Unicode whitespace run to one ASCII space, preserve the first accepted display casing while the row exists, and must contain 1..40 Unicode code points after collapse. `normalizedLabel` is `label.normalize("NFKC").toLowerCase()`; uniqueness uses that value. Duplicate add keeps existing casing; hard delete then re-add uses new casing. Course words are reserved. Full replacement deletes absent custom/course rows first, inserts each desired normalized row with conflict-ignore to preserve the row/casing that exists at execution time, and then applies the recipe update. D1 commit order is the serialization order: simultaneous course replacements are later-commit-wins; full replace committed later replaces the whole set; a later delta add/remove applies to that replacement; two adds observed at nine yield one ten-tag success and one tag_conflict. D1 serializes the transaction against acquisition, so the predicate is true for all statements or false for all, preventing mid-batch expiry from partial writes. After commit, a fresh-time lease check plus source postcheck decides whether this holder may query; expiration after a complete commit returns internal error/no query but leaves one complete index.

A holder may perform at most three fresh fingerprint/build/guarded-atomic-replacement/postcheck cycles. Crash leaves the old complete index or one complete committed index and the lease expires. A matching postcheck is that holder response's consistency point; it may release then query the complete index, while a source commit after that point belongs to the next request. Logged-in no-store projections are immediate; anonymous recipe/search responses retain `public, max-age=60, stale-while-revalidate=300`. Future AI proposals require a separate acceptance model.
